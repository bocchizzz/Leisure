## 1.2 适用范围

本文档适用于赏金布（Campus Quest）校园猎人公会任务悬赏平台的需求分析、系统设计、开发实现、测试验收、部署实施和后续运营维护。

系统主要面向高校学生日常学习、生活、活动协作和轻技能互助场景，支持用户以“委托人”身份发布悬赏任务，也支持用户以“猎人”身份申请接单并完成任务。系统同时提供校园身份认证、任务大厅、任务履约、证据留存、双方评价、猎人等级、信誉分、徽章、榜单、校园小法庭、陪审团投票、AI 书记官辅助、消息通知和后台治理功能。

系统适用场景包括但不限于：

1. 校园内部生活互助：快递代取、物品代买、打印跑腿、宿舍搬运、临时排队等；

2. 校园学习辅助：资料整理、课程笔记归纳、表格整理、学习资源共享协助等；

3. 校园轻技能协作：PPT 美化、海报设计、文案润色、视频剪辑、活动摄影等；

4. 学生组织协作：社团活动协助、场地布置、物资搬运、签到协助、宣传协助等；

5. 校园信用治理：任务评价、信誉沉淀、纠纷证据归档、判例归档、违规记录管理等。

系统不适用于以下场景：

1. 违反法律法规、校规校纪、公序良俗的任务；

2. 代考、代写论文、违规刷课、非法交易、侵犯隐私等高风险任务；

3. 法律意义上的司法裁判、专业仲裁或法律咨询服务；

4. 高金额复杂商业外包、长期雇佣关系或强监管金融服务；

5. 危险品运输、医疗行为、个人敏感信息买卖等高风险场景。

# 2. 总体概述

## 2.1 软件概述

### 2.1.1 产品定位

赏金布（Campus Quest）是一款面向高校学生的校园任务悬赏与信用治理平台。系统以“校园猎人公会”为产品设定，将学生之间零散、临时、轻量的互助需求转化为标准化、可追踪、可评价、可仲裁、可沉淀信用的任务流程。

系统定位不是单纯的信息发布平台，也不是普通跑腿平台，而是一个兼具任务协作、信用成长、趣味激励、争议处理和 AI 辅助能力的校园微协作平台。平台通过任务发布、申请接单、履约举证、确认评价、等级成长、小法庭仲裁、民意投票和管理员治理，提升校园互助的匹配效率、履约透明度和信任稳定性。

平台建设重点包括：

1. 让校园碎片化需求能够被清晰发布；

2. 让有时间、有技能、有行动能力的同学能够可靠接单；

3. 让任务过程具备状态记录、履约证据和完成标准；

4. 让认真履约的用户能够积累等级、信誉和荣誉；

5. 让争议任务能够进入有证据、有投票、有裁决、有归档的处理流程；

6. 让 AI 能力服务于具体任务流程和纠纷处理流程；

7. 让后台管理具备用户治理、任务审核、纠纷处理和数据统计能力。

### 2.1.2 产品介绍

赏金布以“任务发布—申请接单—选择猎人—履约举证—确认完成—双方评价—信用成长”为普通任务主链路；以“发起纠纷—提交证据—AI 摘要—陪审团投票—管理员裁决—判例归档”为争议处理链路。

用户进入平台后，可以完成注册登录和校园身份认证。完成认证后，用户可以发布校园互助任务，也可以进入任务大厅浏览任务并申请接单。委托人可查看申请者的猎人等级、信誉分、完成率、评价记录等信息，并选择合适的猎人。猎人完成任务后，需要提交履约证据；委托人确认完成后，双方进行评价，系统根据任务结果更新经验值、信誉分、徽章和用户画像。

当任务履约过程中发生争议时，任一符合条件的任务参与方可发起校园小法庭。小法庭流程中，双方提交陈述和证据，AI 书记官生成客观案件摘要、争议焦点、证据分析和轻松点评，陪审团参与民意投票，管理员结合任务契约、双方证据、AI 摘要、投票结果和平台规则进行最终裁决。裁决结果影响任务状态、赏金处理、信誉分、经验值和违规记录，并形成判例归档。

平台支持以下核心能力：

1. 用户身份与校园认证；

2. 多类型任务发布；

3. 任务大厅、搜索、筛选、排序；

4. 猎人申请接单；

5. 委托人选择猎人；

6. 任务契约生成；

7. 任务状态流转；

8. 履约证据提交；

9. 确认完成与双方评价；

10. 经验值、等级、信誉分、徽章、榜单；

11. 小法庭立案、证据墙、陪审团投票、管理员裁决；

12. AI 任务拆解、赏金建议、风险提示、证据摘要、案件点评；

13. 消息通知与站内提醒；

14. 后台审核、用户治理、违规记录、数据统计、运营配置。

# 3. 任务概述

## 3.1 目标

系统建设目标如下：

### 3.1.1 业务目标

1. 建立一个面向高校学生的校园任务悬赏平台；

2. 支持学生将临时互助需求发布为标准化任务；

3. 支持学生通过任务大厅寻找适合自己的任务；

4. 建立委托人与猎人之间的任务契约和状态流转机制；

5. 建立任务完成后的评价和信用沉淀机制；

6. 建立任务争议的校园小法庭处理机制；

7. 通过 AI 辅助降低任务发布和纠纷处理成本；

8. 通过后台治理保障任务安全、用户秩序和平台可运营性。

### 3.1.2 功能目标

1. 支持用户注册、登录、校园认证和个人资料维护；

## 3.3 假定和约束

### 3.3.1 业务假定

1. 平台用户以高校学生为主体；

2. 用户具备基础互联网应用使用能力；

3. 用户发布的任务大多为低金额、轻量化、短周期任务；

4. 任务完成质量可以通过文字、图片、截图、文件等方式进行基本证明；

5. 平台小法庭用于平台内部争议处理，不具有法律裁判属性；

6. AI 辅助内容仅作为任务发布、风险识别、证据整理和管理员处理的参考。

### 3.3.2 业务约束

1. 用户必须注册登录后才能进行任务发布、接单、评价和投票；

2. 用户必须完成校园认证后才能发布任务和申请接单；

3. 任务不得违反法律法规、校规校纪和平台任务准入规则；

4. 同一任务同一用户只能提交一次有效接单申请；

5. 委托人不能申请承接自己发布的任务；

6. 猎人不能在任务未被选中时提交履约证据；

7. 任务完成后才能进行评价；

8. 同一任务双方各自最多评价一次；

9. 存在纠纷中的任务不得直接进入普通完成结算流程；

10. AI 不具备最终裁决权限；

11. 管理员裁决必须保留操作记录；

12. 陪审团投票结果只作为管理员裁决参考。

### 3.3.3 技术约束

1. 系统应支持 Web 端访问；

2. 系统应采用前后端分离方式实现；

3. 系统应通过统一入口对外提供服务；

4. 业务接口应采用统一响应格式；

5. 用户身份认证应采用令牌机制；

6. 文件上传应限制文件类型、大小和访问权限；

7. 关键操作应记录审计日志；

8. 系统应具备基础异常处理和错误提示能力；

9. 系统应支持后续模块扩展。

# 4.2.1 用户身份管理

## 4.2.1.1 注册

### 4.2.1.1.1 功能描述

用户通过注册页面创建平台账号。注册成功后，系统生成用户基础档案。用户完成登录后，可继续进行校园身份认证。未完成校园认证的用户只能使用部分浏览类功能，不可发布任务、申请接单、参与投票或发起纠纷。

### 4.2.1.1.2 输入

| 参数名    | 类型    | 是否必填 | 说明      | 校验规则            |
| ------ | ----- | ---- | ------- | --------------- |
| 用户昵称   | 文本输入框 | 是    | 平台展示名称  | 2—20 个字符，不得含敏感词 |
| 手机号    | 文本输入框 | 是    | 登录和验证使用 | 格式正确，系统内唯一      |
| 验证码    | 文本输入框 | 是    | 手机验证码   | 与发送记录一致，未过期     |
| 邮箱     | 文本输入框 | 否    | 可用于找回密码 | 格式正确            |
| 学校     | 下拉选择  | 是    | 所属高校    | 必须在学校字典中        |
| 学号     | 文本输入框 | 否    | 校园认证使用  | 认证时必填           |
| 密码     | 密码输入框 | 是    | 登录密码    | 8—20 位，包含数字和字母  |
| 确认密码   | 密码输入框 | 是    | 二次确认    | 必须与密码一致         |
| 用户协议勾选 | 复选框   | 是    | 同意用户协议  | 必须勾选            |
| 隐私政策勾选 | 复选框   | 是    | 同意隐私政策  | 必须勾选            |
| 注册     | 按钮    | 是    | 提交注册    | 点击后触发注册         |
| 取消     | 按钮    | 否    | 放弃注册    | 返回登录页           |

### 4.2.1.1.3 业务流程

1. 用户进入注册页面；

2. 用户填写昵称、手机号、验证码、学校、密码等信息；

3. 用户勾选用户协议和隐私政策；

4. 用户点击“注册”；

5. 系统校验手机号、验证码、密码规则和唯一性；

6. 校验通过后系统创建用户账号；

7. 系统提示注册成功；

8. 用户跳转到登录页面或自动进入登录状态；

9. 系统引导用户进行校园身份认证。

### 4.2.1.1.4 业务规则

1. 手机号必须唯一；

2. 昵称不得包含违法违规、侮辱性、广告性词语；

3. 密码不得与手机号完全相同；

4. 验证码 60 秒内不可重复发送；

5. 验证码有效期默认为 5 分钟；

6. 同一手机号每日验证码发送次数应有限制；

7. 用户未同意协议和隐私政策时不得注册；

8. 注册成功后默认角色为普通用户；

9. 注册成功后默认信誉分为 100 分；

10. 注册成功后默认经验值为 0；

11. 注册成功后默认等级为 Lv0 见习猎人；

12. 注册成功后账号状态为正常。

### 4.2.1.1.5 使用者

游客、普通用户。

### 4.2.1.1.6 界面原型与功能

注册页面应包含以下区域：

1. 平台 Logo 与平台名称；

2. 注册信息表单；

3. 手机验证码发送按钮；

4. 用户协议与隐私政策勾选区域；

5. 注册按钮；

6. 返回登录按钮；

7. 错误提示区域。

## 4.2.1.2 登录

### 4.2.1.2.1 功能描述

用户通过手机号和密码登录平台。登录成功后，系统返回身份令牌和用户基础信息，并根据用户角色跳转到对应页面。

### 4.2.1.2.2 输入

| 参数名    | 类型    | 是否必填 | 说明         |
| ------ | ----- | ---- | ---------- |
| 手机号    | 文本输入框 | 是    | 注册手机号      |
| 密码     | 密码输入框 | 是    | 用户密码       |
| 记住登录状态 | 复选框   | 否    | 延长本地登录保持时间 |
| 登录     | 按钮    | 是    | 提交登录       |
| 忘记密码   | 链接    | 否    | 跳转找回密码     |
| 注册账号   | 链接    | 否    | 跳转注册页面     |

### 4.2.1.2.3 业务流程

1. 用户进入登录页面；

2. 输入手机号和密码；

3. 点击“登录”；

4. 系统校验账号是否存在；

5. 系统校验密码是否正确；

6. 系统校验账号状态是否正常；

7. 校验通过后生成登录令牌；

8. 系统返回用户信息和权限信息；

9. 前端保存令牌并跳转首页。

### 4.2.1.2.4 业务规则

1. 账号不存在时提示账号或密码错误；

2. 密码错误时提示账号或密码错误；

3. 账号被封禁时禁止登录并提示封禁原因；

4. 连续输错密码达到限制次数时，短时间内限制登录；

5. 登录成功后更新最近登录时间；

6. 登录令牌过期后需要重新登录；

7. 管理员登录后可进入后台管理入口。

### 4.2.1.2.5 使用者

普通用户、委托人、猎人、陪审团、管理员、超级管理员。

# 4.2.3 任务发布

## 4.2.3.1 发布悬赏任务

### 4.2.3.1.1 功能描述

委托人通过任务发布页面填写任务标题、分类、描述、赏金、地点、截止时间、完成标准等信息。系统支持 AI 辅助拆解任务内容、推荐分类、建议赏金和提示风险。

### 4.2.3.1.2 输入

| 参数名    | 类型      | 是否必填 | 说明             | 校验规则        |
| ------ | ------- | ---- | -------------- | ----------- |
| 任务标题   | 文本输入框   | 是    | 简要描述任务         | 5—40 个字符    |
| 任务分类   | 下拉选择    | 是    | 跑腿、学习、设计、活动等   | 必须来自分类字典    |
| 任务描述   | 文本域     | 是    | 详细说明需求         | 20—1000 个字符 |
| 任务地点   | 文本输入框   | 否    | 校内地点或线上        | 线上任务可为空     |
| 赏金类型   | 单选项     | 是    | 积分、虚拟赏金、线下约定等  | 按平台规则配置     |
| 赏金数额   | 数字输入框   | 是    | 任务价值           | 大于 0，不能超过上限 |
| 截止时间   | 日期时间选择器 | 是    | 任务完成期限         | 必须晚于当前时间    |
| 任务难度   | 单选项     | 否    | 简单、普通、困难       | 可由 AI 推荐    |
| 完成标准   | 文本域     | 是    | 验收标准           | 不得为空        |
| 证据要求   | 多选项     | 是    | 图片、截图、文件、文字说明等 | 至少选择一项      |
| 附件     | 文件上传    | 否    | 补充材料           | 限制格式和大小     |
| 是否需要审核 | 系统判断    | 否    | 根据规则判断         | 高风险任务需审核    |
| AI 拆解  | 按钮      | 否    | 调用 AI 辅助       |             |
| 保存草稿   | 按钮      | 否    | 保存为草稿          |             |
| 发布任务   | 按钮      | 是    | 提交发布           |             |

### 4.2.3.1.3 业务流程

1. 委托人进入发布任务页面；

2. 填写任务基本信息；

3. 可点击“AI 拆解”生成任务步骤、分类建议、赏金建议和风险提示；

4. 委托人确认或修改 AI 建议；

5. 系统校验必填项、敏感词、赏金范围、截止时间；

6. 系统判断是否需要人工审核；

7. 无需审核的任务进入招募中状态；

8. 需要审核的任务进入待审核状态；

9. 系统向委托人发送发布结果通知；

10. 任务公开后进入任务大厅展示。

### 4.2.3.1.4 业务规则

1. 只有已登录且已完成校园认证的用户可以发布任务；

2. 任务标题和描述不得包含违法违规内容；

3. 任务不得涉及代考、代写、刷课、违规交易、侵犯隐私等内容；

4. 截止时间必须晚于当前时间；

5. 赏金数额必须在系统配置范围内；

6. 任务发布后，关键字段修改应受限制；

7. 已有猎人申请后，委托人修改赏金、截止时间、完成标准时需提示申请者；

8. 任务进入进行中后，委托人不得随意取消；

9. 高风险关键词命中的任务进入人工审核；

10. AI 建议仅供参考，最终发布内容由委托人确认；

11. 发布任务后系统生成任务编号；

12. 任务发布行为写入操作日志。

### 4.2.3.1.5 使用者

委托人、管理员、AI 书记官。

### 4.2.3.1.6 界面原型与功能

任务发布页面应包含：

1. 任务基础信息区；

2. 任务分类与难度区；

3. 赏金与截止时间区；

4. 完成标准与证据要求区；

5. 附件上传区；

6. AI 辅助区；

7. 风险提示区；

8. 保存草稿和发布按钮。

### 4.2.3.1.7 输出

| 输出类型   | 输出内容                            |
| ------ | ------------------------------- |
| 成功     | 任务发布成功，返回任务编号和任务状态              |
| 待审核    | 任务提交成功，等待管理员审核                  |
| 失败     | 发布失败并提示原因                       |
| 常见失败原因 | 未认证、字段缺失、赏金不合法、截止时间错误、内容违规、附件超限 |

# 4.2.7 任务履约与证据提交

## 4.2.7.1 提交履约证据

### 4.2.7.1.1 功能描述

被选中的猎人在完成任务后，根据任务要求提交履约证据。履约证据可包括文字说明、图片、截图、文件、链接等。提交后任务进入待确认状态。

### 4.2.7.1.2 输入

| 参数名  | 类型   | 是否必填  | 说明      |
| ---- | ---- | ----- | ------- |
| 任务编号 | 系统参数 | 是     | 对应任务    |
| 履约说明 | 文本域  | 是     | 说明完成情况  |
| 证据图片 | 文件上传 | 视任务要求 | 图片证明    |
| 证据文件 | 文件上传 | 视任务要求 | 文档、压缩包等 |
| 完成时间 | 系统记录 | 是     | 提交时间    |
| 补充备注 | 文本域  | 否     | 额外说明    |
| 提交证据 | 按钮   | 是     | 提交履约材料  |

### 4.2.7.1.3 业务流程

1. 猎人进入进行中的任务详情页；

2. 点击“提交履约证据”；

3. 填写履约说明；

4. 上传符合要求的证据材料；

5. 系统校验文件类型、大小和必填项；

6. 系统保存履约证据；

7. 任务状态更新为待确认；

8. 系统通知委托人确认；

9. 委托人查看履约证据；

10. 委托人确认完成、要求补充或发起纠纷。

### 4.2.7.1.4 业务规则

1. 只有被选中猎人可以提交履约证据；

2. 任务必须处于进行中状态；

3. 证据类型应符合任务发布时的证据要求；

4. 图片和文件应进行大小和格式限制；

5. 猎人可以在委托人确认前补充证据；

6. 委托人确认后，履约证据进入归档状态；

7. 证据涉及隐私时仅任务双方和管理员可见；

8. 履约提交时间用于判断是否超时；

9. 逾期提交应标记为超时；

10. 证据删除或替换应保留历史记录。

### 4.2.7.1.5 使用者

猎人、委托人、管理员。

### 4.2.7.1.6 输出

| 输出类型   | 输出内容                          |
| ------ | ----------------------------- |
| 成功     | 履约证据提交成功，任务进入待确认              |
| 失败     | 提交失败并提示原因                     |
| 常见失败原因 | 非任务猎人、任务状态错误、证据缺失、文件格式错误、文件过大 |

# 4.2.8 任务确认与双方评价

## 4.2.8.1 确认完成

### 4.2.8.1.1 功能描述

委托人查看猎人提交的履约证据后，可确认任务完成。确认后任务进入已完成状态，并开启双方评价入口。

### 4.2.8.1.2 输入

| 参数名  | 类型   | 是否必填 | 说明             |
| ---- | ---- | ---- | -------------- |
| 任务编号 | 系统参数 | 是    | 对应任务           |
| 确认结果 | 单选   | 是    | 确认完成、要求补充、发起纠纷 |
| 确认说明 | 文本域  | 否    | 补充意见           |
| 确认按钮 | 按钮   | 是    | 提交确认操作         |

### 4.2.8.1.3 业务流程

1. 委托人收到待确认通知；

2. 打开任务详情页；

3. 查看猎人履约说明和证据；

4. 选择确认完成；

5. 系统更新任务状态为已完成；

6. 系统生成评价入口；

7. 系统更新任务完成记录；

8. 系统通知猎人。

### 4.2.8.1.4 业务规则

1. 只有任务委托人可以确认完成；

2. 任务必须处于待确认状态；

3. 确认完成后不可直接撤回；

4. 确认完成后系统更新猎人的完成任务数；

5. 若超过确认期限未处理，系统可根据配置自动提醒或进入超时待处理；

6. 委托人认为未完成时，可要求补充或发起纠纷；

7. 任务确认后双方可评价；

8. 评价完成后影响信誉分和经验值。

### 4.2.8.1.5 使用者

委托人。

### 4.2.8.1.6 输出

| 输出类型   | 输出内容                     |
| ------ | ------------------------ |
| 成功     | 任务确认完成                   |
| 失败     | 确认失败并提示原因                |
| 常见失败原因 | 非委托人、任务状态错误、任务已进入纠纷、系统异常 |

## 4.2.8.2 双方评价

### 4.2.8.2.1 功能描述

任务完成后，委托人和猎人可互相进行评分和文字评价。评价用于形成用户信用画像，并影响信誉分、等级成长和后续推荐。

### 4.2.8.2.2 输入

| 参数名    | 类型    | 是否必填 | 说明          |
| ------ | ----- | ---- | ----------- |
| 任务编号   | 系统参数  | 是    | 对应任务        |
| 评分     | 星级/数字 | 是    | 1—5 分       |
| 评价标签   | 多选    | 否    | 准时、沟通好、质量高等 |
| 文字评价   | 文本域   | 否    | 评价内容        |
| 是否匿名展示 | 开关    | 否    | 前台是否匿名      |
| 提交评价   | 按钮    | 是    | 提交评价        |

### 4.2.8.2.3 业务流程

1. 任务完成后，系统向双方开放评价入口；

2. 用户填写评分、标签和文字评价；

3. 系统校验评价内容；

4. 提交后保存评价记录；

5. 系统更新被评价方评分、好评率和信誉分；

6. 系统判断是否触发徽章；

7. 系统关闭该用户对该任务的评价入口。

### 4.2.8.2.4 业务规则

1. 只有任务双方可以评价；

2. 每个用户对同一任务只能评价一次；

3. 任务未完成不得评价；

4. 评价内容不得包含辱骂、广告、隐私泄露等内容；

5. 评分低于设定阈值时，可提示用户是否发起纠纷；

6. 恶意评价可由管理员处理；

7. 评价提交后原则上不可修改；

8. 评价影响信誉分；

9. 高质量好评可增加经验值；

10. 系统应保留评价提交时间。

### 4.2.8.2.5 使用者

委托人、猎人、管理员。

### 4.2.8.2.6 输出

| 输出类型   | 输出内容                 |
| ------ | -------------------- |
| 成功     | 评价提交成功               |
| 失败     | 评价失败并提示原因            |
| 常见失败原因 | 已评价、任务未完成、无评价权限、内容违规 |

# 4.2.9 猎人等级、信誉分与徽章体系

## 4.2.9.1 经验值与等级

### 4.2.9.1.1 功能描述

系统根据用户完成任务、获得好评、准时履约、参与仲裁投票等行为计算经验值，并根据经验值确定猎人等级。等级用于展示成长身份，也影响任务推荐、投票资格和投票权重。

### 4.2.9.1.2 等级规则

| 等级  | 称号   | 经验范围      | 权益               |
| --- | ---- | --------- | ---------------- |
| Lv0 | 见习猎人 | 0—99      | 可浏览任务，可申请普通任务    |
| Lv1 | 铁牌猎人 | 100—299   | 可参与基础任务，具备投票申请资格 |
| Lv2 | 铜牌猎人 | 300—699   | 任务推荐权重提升         |
| Lv3 | 银牌猎人 | 700—1499  | 可申请高赏金任务         |
| Lv4 | 金牌猎人 | 1500—2999 | 投票权重提升，优先展示      |
| Lv5 | 星徽猎人 | 3000—5999 | 可进入高阶榜单          |
| Lv6 | 传奇猎人 | 6000 以上   | 可成为荣誉陪审员候选       |

### 4.2.9.1.3 经验值计算规则

经验值基础计算公式如下：

```text
XP = 10 × 完成任务数
   + 15 × 好评数
   + 20 × 被采纳的仲裁投票数
   - 10 × 超时次数
   - 30 × 违规次数
   - 50 × 恶意纠纷次数
```

系统可根据运营配置调整各项权重，但调整应保留版本记录。

### 4.2.9.1.4 业务规则

1. 经验值不得低于 0；

2. 完成任务后增加基础经验值；

3. 获得好评后增加额外经验值；

4. 恶意违规行为扣减经验值；

5. 等级根据经验值自动计算；

6. 等级变化时系统发送通知；

7. 管理员可查看经验变动记录；

8. 管理员不得随意修改经验值，特殊修正必须记录原因；

9. 经验值用于成长，不等同于信誉分。

---

## 4.2.9.2 信誉分

### 4.2.9.2.1 功能描述

信誉分用于衡量用户可信程度。信誉分可随良好履约行为上升，也可因差评、超时、违规、败诉纠纷下降。信誉分影响用户任务申请、任务发布、投票资格和后台风险提示。

### 4.2.9.2.2 信誉分计算规则

信誉分基础计算公式如下：

```text
信誉分 = 100
      + 2 × 好评数
      + 1 × 准时完成数
      - 5 × 差评数
      - 8 × 超时数
      - 15 × 败诉纠纷数
      - 30 × 严重违规数
```

### 4.2.9.2.3 信誉等级

| 信誉分范围   | 信誉等级 | 系统处理        |
| ------- | ---- | ----------- |
| 120 以上  | 优秀   | 优先推荐、榜单加权   |
| 100—119 | 良好   | 正常使用        |
| 80—99   | 一般   | 正常使用，部分风险提示 |
| 60—79   | 较低   | 接单申请可能受限    |
| 40—59   | 风险   | 限制高赏金任务和投票  |
| 40 以下   | 严重风险 | 禁止接单，进入人工审核 |

### 4.2.9.2.4 业务规则

1. 新用户默认信誉分为 100；

2. 信誉分可升可降；

3. 严重违规可触发信誉分大幅扣减；

4. 信誉分过低时限制部分权限；

5. 管理员裁决可影响信誉分；

6. 信誉分变化应记录来源事件；

7. 用户可查看自己的信誉分变动明细；

8. 他人仅能查看公开信誉等级或分数摘要。

## 4.2.9.3 徽章与榜单

### 4.2.9.3.1 功能描述

系统根据用户行为授予徽章，并按任务完成数、好评率、信誉分、经验值等指标形成榜单，增强用户参与感和成长激励。

### 4.2.9.3.2 徽章类型

| 徽章名称  | 获得条件            | 说明      |
| ----- | --------------- | ------- |
| 准时达人  | 连续 10 次准时完成任务   | 鼓励守时    |
| 好评猎人  | 获得 20 次以上好评     | 鼓励高质量履约 |
| 公会新星  | 新用户完成 3 个任务且无差评 | 鼓励新用户成长 |
| 金牌跑腿  | 跑腿类任务完成数达到阈值    | 分类专长    |
| 设计能手  | 设计类任务完成数达到阈值    | 分类专长    |
| 公正陪审员 | 投票多次与最终裁决一致     | 鼓励理性投票  |
| 和解推动者 | 多次案件投票选择和解且被采纳  | 鼓励友好解决  |

### 4.2.9.3.3 榜单类型

| 榜单    | 排序依据          |
| ----- | ------------- |
| 猎人总榜  | 经验值、信誉分、完成数综合 |
| 好评榜   | 好评率和好评数量      |
| 准时榜   | 准时完成率         |
| 新星榜   | 新用户成长速度       |
| 陪审团榜  | 投票采纳率和参与次数    |
| 分类专长榜 | 分类任务完成数量和评分   |

### 4.2.9.3.4 业务规则

1. 徽章授予应自动触发；

2. 用户严重违规时可隐藏部分荣誉展示；

3. 榜单应过滤封禁用户；

4. 榜单可按周、月、总榜展示；

5. 管理员可配置部分徽章阈值；

6. 徽章撤销必须记录原因。

# 4.2.10 校园小法庭仲裁

## 4.2.10.1 发起纠纷

### 4.2.10.1.1 功能描述

当任务出现履约争议、质量争议、超时争议、报酬争议、证据争议、恶意任务或恶意接单时，任务参与方可发起校园小法庭。小法庭用于平台内部争议处理，不具有法律裁判属性。

### 4.2.10.1.2 输入

| 参数名  | 类型    | 是否必填 | 说明              |
| ---- | ----- | ---- | --------------- |
| 任务编号 | 系统参数  | 是    | 对应任务            |
| 纠纷类型 | 下拉选择  | 是    | 履约、质量、超时、报酬、证据等 |
| 发起理由 | 文本域   | 是    | 纠纷原因            |
| 诉求   | 多选/文本 | 是    | 要求补做、部分确认、取消任务等 |
| 初始证据 | 文件上传  | 否    | 图片、截图、文件等       |
| 提交立案 | 按钮    | 是    | 发起小法庭           |

### 4.2.10.1.3 业务流程

1. 用户进入任务详情页；

2. 点击“发起纠纷”；

3. 选择纠纷类型；

4. 填写发起理由和诉求；

5. 上传初始证据；

6. 系统校验任务是否允许立案；

7. 系统生成纠纷案件编号；

8. 任务状态更新为纠纷中；

9. 系统通知另一方提交陈述和证据；

10. 案件进入待陈述或待举证状态。

### 4.2.10.1.4 业务规则

1. 只有任务双方可以发起纠纷；

2. 任务必须处于可争议状态；

3. 已关闭任务超过申诉期限后不得发起纠纷；

4. 同一任务同一阶段只能存在一个进行中的纠纷案件；

5. 发起纠纷后任务进入纠纷中状态；

6. 纠纷期间普通确认完成流程暂停；

7. 恶意发起纠纷可扣减信誉分；

8. 管理员可驳回不符合规则的立案；

9. 所有纠纷材料应归档保存；

10. 纠纷流程中应提示用户文明表达。

### 4.2.10.1.5 使用者

委托人、猎人、管理员。

### 4.2.10.1.6 输出

| 输出类型   | 输出内容                        |
| ------ | --------------------------- |
| 成功     | 纠纷案件创建成功                    |
| 失败     | 立案失败并提示原因                   |
| 常见失败原因 | 非任务参与方、任务状态不可申诉、超过申诉期限、重复立案 |

---

## 4.2.10.2 双方陈述与证据墙

### 4.2.10.2.1 功能描述

纠纷案件创建后，双方可在规定时间内提交陈述和证据。系统将证据按提交方、时间、类型形成证据墙，供 AI 书记官、陪审团和管理员查看。

### 4.2.10.2.2 输入

| 参数名  | 类型   | 是否必填 | 说明             |
| ---- | ---- | ---- | -------------- |
| 案件编号 | 系统参数 | 是    | 对应纠纷案件         |
| 陈述内容 | 文本域  | 是    | 当事人说明          |
| 证据类型 | 下拉选择 | 是    | 图片、截图、文件、聊天记录等 |
| 证据文件 | 文件上传 | 否    | 支撑材料           |
| 证据说明 | 文本域  | 否    | 对证据的解释         |
| 提交   | 按钮   | 是    | 提交陈述或证据        |

### 4.2.10.2.3 业务流程

1. 当事人收到举证通知；

2. 进入案件详情页；

3. 填写陈述内容；

4. 上传证据文件；

5. 系统校验证据格式；

6. 系统保存证据并生成证据墙；

7. 双方举证结束或时间到期后进入 AI 摘要阶段。

### 4.2.10.2.4 业务规则

1. 双方均可提交陈述和证据；

2. 证据提交应限定时间；

3. 超时未提交视为放弃补充；

4. 证据文件不得包含违法违规内容；

5. 证据涉及隐私时，陪审团只查看脱敏摘要；

6. 管理员可隐藏不适宜公开展示的证据；

7. 证据提交后不得直接删除；

8. 补充证据应保留版本；

9. 系统记录提交时间和提交人；

10. AI 摘要应基于双方已提交材料生成。

### 4.2.10.2.5 使用者

委托人、猎人、管理员、AI 书记官、陪审团。

### 4.2.10.2.6 输出

| 输出内容 | 说明         |
| ---- | ---------- |
| 双方陈述 | 按提交方展示     |
| 证据列表 | 按时间排序      |
| 证据类型 | 图片、文件、文字等  |
| 证据说明 | 用户填写说明     |
| 提交时间 | 系统记录       |
| 脱敏状态 | 是否对陪审团脱敏展示 |

---

## 4.2.10.3 管理员裁决

### 4.2.10.3.1 功能描述

管理员查看任务契约、双方陈述、证据墙、AI 案件摘要、陪审团投票结果和平台规则后，对纠纷案件作出处理决定。

### 4.2.10.3.2 输入

| 参数名    | 类型   | 是否必填 | 说明                         |
| ------ | ---- | ---- | -------------------------- |
| 案件编号   | 系统参数 | 是    | 待裁决案件                      |
| 裁决结果   | 下拉选择 | 是    | 支持委托人、支持猎人、部分支持、建议和解、不予受理等 |
| 任务处理方式 | 下拉选择 | 是    | 完成、取消、部分完成、要求补做等           |
| 赏金处理   | 下拉选择 | 是    | 全额、部分、不发放、线下处理等            |
| 信誉处理   | 表单   | 否    | 扣分、加分、警告等                  |
| 违规记录   | 表单   | 否    | 是否记录违规                     |
| 裁决说明   | 文本域  | 是    | 裁决理由                       |
| 保存裁决   | 按钮   | 是    | 提交裁决                       |

### 4.2.10.3.3 业务流程

1. 管理员进入后台纠纷处理页面；

2. 查看待裁决案件列表；

3. 打开案件详情；

4. 查看任务契约、双方陈述、证据墙；

5. 查看 AI 摘要和争议焦点；

6. 查看陪审团投票统计；

7. 选择裁决结果和任务处理方式；

8. 填写裁决说明；

9. 提交裁决；

10. 系统更新案件状态为已裁决；

11. 系统更新任务状态、信誉分、经验值和违规记录；

12. 系统通知任务双方；

13. 系统生成判例归档。

### 4.2.10.3.4 业务规则

1. 只有管理员可以裁决；

2. 管理员不得裁决自己参与的任务；

3. 裁决必须填写理由；

4. 裁决结果应与任务处理方式一致；

5. 信誉分调整必须明确来源；

6. 违规记录必须绑定具体案件；

9. 投票结果不直接决定裁决；

10. AI 摘要不直接决定裁决；

11. 裁决后案件进入归档流程；

12. 裁决结果应通知双方。

### 4.2.10.3.5 使用者

管理员、超级管理员。

### 4.2.10.3.6 输出

| 输出类型 | 输出内容                     |

# 4.2.11 陪审团投票与民意统计

## 4.2.11.1 陪审团投票

### 4.2.11.1.1 功能描述

符合条件的用户可作为陪审团参与纠纷案件投票。投票选项包括支持委托人、支持猎人、证据不足、建议和解。系统根据用户等级、信誉和历史投票采纳情况计算投票权重。

### 4.2.11.1.2 输入

| 参数名  | 类型   | 是否必填 | 说明     |
| ---- | ---- | ---- | ------ |
| 案件编号 | 系统参数 | 是    | 投票案件   |
| 投票选项 | 单选   | 是    | 四类投票选项 |
| 投票理由 | 文本域  | 否    | 简要说明   |
| 提交投票 | 按钮   | 是    | 提交投票   |

### 4.2.11.1.3 投票选项

| 选项    | 含义          |
| ----- | ----------- |
| 支持委托人 | 认为委托人诉求更合理  |
| 支持猎人  | 认为猎人履约更合理   |
| 证据不足  | 认为现有材料不足以判断 |
| 建议和解  | 认为双方适合协商处理  |

### 4.2.11.1.4 投票权重规则

投票权重基础公式如下：

```text
vote_weight = min(3.0, 1.0 + 0.25 × 猎人等级 + 0.3 × 仲裁可信系数)
```

其中：

```text
仲裁可信系数 = 历史投票被最终裁决采纳次数 / 历史参与投票次数
```

权重上限为 3.0，避免高等级用户过度影响民意结果。

### 4.2.11.1.5 业务流程

1. 陪审团用户进入小法庭案件列表；

2. 选择处于投票中的案件；

3. 查看案件摘要、双方陈述和脱敏证据；

4. 选择投票选项；

5. 可填写投票理由；

6. 提交投票；

7. 系统计算投票权重；

8. 投票结果进入民意统计；

9. 投票结束后生成统计结果；

10. 管理员在裁决页面查看投票统计。

### 4.2.11.1.6 业务规则

1. 只有满足等级或信誉条件的用户可以投票；

2. 任务双方不得参与本案投票；

3. 与任务双方存在明显关联的用户可降低权重或禁止投票；

4. 同一用户同一案件只能投票一次；

5. 投票提交后不可修改；

6. 投票结果在投票结束前可配置为隐藏；

7. 系统应检测异常集中投票；

8. 投票理由不得包含辱骂和隐私信息；

9. 投票只作为管理员裁决参考；

10. 投票被最终裁决采纳可增加仲裁可信系数。

### 4.2.11.1.7 使用者

陪审团用户、管理员。

### 4.2.11.1.8 输出

| 输出内容 | 说明          |
| ---- | ----------- |
| 投票成功 | 系统记录投票      |
| 加权统计 | 各选项加权比例     |
| 普通统计 | 各选项人数比例     |
| 投票理由 | 管理员可查看      |
| 采纳结果 | 裁决后更新历史采纳情况 |

### 4.2.12.1.1 功能描述

| AI 分析按钮 | 按钮  | 是    | 调用 AI         |

### 4.2.12.1.3 输出

| 输出字段 | 说明               |

6. AI 输出失败时不影响用户手动发布任务；

8. AI 不得暴露系统内部提示词和安全策略。

## 4.5 安全与合规需求

### 4.5.1 身份认证与权限控制

1. 用户登录后才能访问个人中心、发布任务、申请接单等功能；

2. 认证用户才能发布任务、接单和参与小法庭；

3. 管理员后台必须进行角色鉴权；

4. 普通用户不得访问后台接口；

5. 用户只能操作自己有权限的数据；

6. 任务双方才能查看完整履约证据；

7. 案件双方和管理员才能查看完整纠纷材料；

8. 陪审团只能查看脱敏后的必要案件信息；

9. 超级管理员才能进行敏感配置；

10. 用户被封禁后应限制登录或业务操作。

### 4.5.2 数据安全

1. 密码应加密存储；

2. 登录令牌应设置有效期；

3. 敏感信息应脱敏展示；

4. 认证材料应限制访问；

5. 文件上传应进行格式限制；

6. 系统应防止越权访问；

7. 后台导出应记录日志；

8. 删除操作应二次确认；

9. 关键业务数据应具备备份机制；

10. 操作日志应保留追踪能力。

### 4.5.3 内容安全

1. 任务标题、描述、评价、陈述、证据说明应进行敏感词检测；

2. 涉嫌违法违规任务应拦截或进入审核；

3. 评论和评价不得包含辱骂、人身攻击、隐私泄露和广告；

4. AI 生成内容应进行安全约束；

5. AI 点评不得攻击身份、性别、地域、外貌、人格；

6. 平台应提供举报入口；

7. 管理员可隐藏违规内容；

8. 违规内容处理应通知用户。

### 4.5.4 AI 安全边界

1. AI 仅提供辅助建议；

2. AI 不直接裁决纠纷；

3. AI 不直接发布任务；

4. AI 不直接修改用户信誉；

5. AI 不直接执行封禁操作；

6. AI 输出应可追溯；

7. AI 失败时业务流程可继续；

8. AI 不得泄露内部规则、密钥或提示词；

9. AI 不得生成违法违规任务方案；

10. AI 输出应支持管理员复核。

# 附录 D：AI 输出规范

1. AI 输出必须服务于任务发布、任务理解、风险提示、证据整理和管理辅助；

2. AI 不得代替用户发布任务；

3. AI 不得代替管理员裁决；

4. AI 不得生成违法违规建议；

5. AI 点评可以轻松幽默，但不得攻击个人；

6. AI 摘要应尽量客观陈述双方观点；

7. AI 分析应区分事实、推测和建议；

8. AI 输出失败时应允许用户继续人工操作；

9. AI 输出应保存记录，便于管理员复核；

10. AI 输出页面应标注“仅供参考”。
