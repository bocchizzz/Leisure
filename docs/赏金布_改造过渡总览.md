# 赏金布改造过渡总览

版本：v0.1  
日期：2026-06-30  
适用项目：`services/campus-market-parent`

## 1. 文档目的

本文档用于连接原“校园二手商城”项目与新目标“赏金布（Campus Quest）校园接单平台”。它不替代《软件需求 - AI.md》，而是回答后续开发最容易卡住的几个过渡问题：

1. 旧系统里的商品、订单、申诉、评价分别迁移成什么业务概念；
2. 现有微服务模块哪些可以复用，哪些需要改造，哪些需要新增；
3. 前后端联调时应优先对齐哪些主链路；
4. 后续开发如何分阶段推进，避免一次性重写导致主业务断裂。

本轮改造文档只做设计与对接准备，不修改 Java 代码、SQL 脚本和既有文档。

## 2. 项目新定位

赏金布（Campus Quest）定位为面向高校学生的校园任务悬赏与接单平台。用户可以作为委托人发布任务，也可以作为猎人申请接单并提交履约证据。任务完成后双方评价，系统沉淀猎人等级、信誉分和任务记录；发生争议时进入校园小法庭，由 AI 书记官辅助摘要、陪审团投票、管理员最终裁决。

核心主链路：

```text
注册登录
-> 完善资料/校园认证
-> 发布悬赏任务
-> 任务大厅展示
-> 猎人申请接单
-> 委托人选择猎人
-> 生成任务契约
-> 猎人提交履约证据
-> 委托人确认完成
-> 双方评价
-> 更新经验、等级、信誉和消息
```

纠纷链路：

```text
任务双方发生争议
-> 发起小法庭案件
-> 双方提交陈述与证据
-> AI 书记官生成案件摘要与点评
-> 陪审团投票
-> 管理员裁决
-> 更新任务结果、信誉、经验、违规记录
-> 归档判例
```

## 3. 现有系统基础

当前项目是 Spring Cloud 微服务结构，主要模块如下：

| 现有模块 | 当前职责 | 赏金布过渡职责 | 处理策略 |
| --- | --- | --- | --- |
| `cm-gateway` | 网关、路由、鉴权前置校验、CORS | 继续作为统一入口，新增任务、小法庭、猎人相关路由 | 可复用，需补充路由 |
| `cm-auth` | 登录注册、用户资料、角色、信誉分 | 用户身份、校园认证、猎人档案、等级、信誉、徽章 | 可复用并扩展 |
| `cm-product` | 商品发布、商品列表、收藏、浏览、文件上传、AI 商品能力 | 任务发布、任务大厅、任务详情、任务收藏/浏览、文件上传、AI 任务辅助 | 重点改造 |
| `cm-trade` | 订单、订单状态、评价 | 接单申请、任务契约、履约证据、确认完成、双方评价 | 重点改造 |
| `cm-notify` | 站内消息、申诉、审计、后台看板 | 消息通知、小法庭初版、案件投票、裁决、审计 | 复用并扩展 |
| `cm-common` | 统一响应、异常、JWT、内部调用拦截 | 保持公共能力，补充通用枚举/安全工具 | 可复用 |

过渡期不强制物理重命名模块。建议先在代码中保持 `cm-product/cm-trade` 等模块名，业务接口逐步迁移到 `/api/tasks`、`/api/contracts`、`/api/court-cases` 等新路径。等主链路稳定后，再评估是否重命名模块。

## 4. 旧概念到新概念映射

| 旧商城概念 | 新赏金布概念 | 说明 |
| --- | --- | --- |
| 商品 `Product` | 悬赏任务 `Task` | 商品标题、描述、价格可迁移为任务标题、任务描述、赏金 |
| 商品列表 | 任务大厅 | 保留搜索、分类、分页、排序能力 |
| 商品详情 | 任务详情 | 增加委托人、任务地点、截止时间、完成标准、申请列表 |
| 商品分类 `ProductCategory` | 任务分类 `TaskCategory` | 从数码/教材等改为跑腿、学习、设计、文案、线上协作等 |
| 商品成色 `ProductCondition` | 任务难度/证据要求 | 不再表示新旧程度，可转为任务难度或履约要求 |
| 价格 `price` | 赏金 `bountyAmount` | 可为积分、虚拟赏金或线下约定金额 |
| 卖家 `sellerId` | 委托人 `publisherId` | 发布任务的人 |
| 买家 `buyerId` | 猎人 `hunterId` | 接单并完成任务的人 |
| 订单 `Order` | 接单申请/任务契约 | 订单需要拆成申请与已选中后的契约 |
| 发货 `ship` | 提交履约证据 | 猎人提交图片、文字、文件等完成证明 |
| 收货完成 `complete` | 委托人确认完成 | 委托人确认任务已完成 |
| 单向评价 | 双方评价 | 委托人与猎人均可评价对方 |
| 申诉 `Appeal` | 小法庭案件 `CourtCase` | 从普通申诉升级为任务纠纷处理流程 |
| 收藏商品 | 收藏任务 | 可保留为任务收藏 |
| 浏览历史 | 任务浏览历史 | 可保留为任务浏览记录 |
| 商品 AI 描述/估价 | AI 任务拆解/赏金建议/风险提示 | AI 能力从商品文案迁移到任务理解 |

## 5. 状态迁移

### 5.1 商品状态到任务状态

| 原商品状态 | 过渡任务状态 | 说明 |
| --- | --- | --- |
| `AVAILABLE` | `PUBLISHED` | 已发布，可被猎人申请 |
| `RESERVED` | `ASSIGNED` | 已选中猎人，进入履约前后 |
| `SOLD` | `COMPLETED` | 任务完成并确认 |
| `REMOVED` | `CANCELLED` | 任务取消、下架或违规移除 |

建议后续完整任务状态：

```text
DRAFT
PENDING_REVIEW
PUBLISHED
APPLYING
ASSIGNED
IN_PROGRESS
WAIT_CONFIRM
COMPLETED
DISPUTED
COURT_REVIEW
RULED
CANCELLED
```

### 5.2 订单状态到接单履约状态

| 原订单状态 | 过渡接单/履约状态 | 说明 |
| --- | --- | --- |
| `PENDING` | `APPLIED` | 猎人已提交申请，等待委托人处理 |
| `CONFIRMED` | `IN_PROGRESS` | 委托人已选择猎人，任务执行中 |
| `SHIPPED` | `WAIT_CONFIRM` | 猎人已提交履约证据，等待委托人确认 |
| `COMPLETED` | `COMPLETED` | 任务确认完成 |
| `CANCELLED` | `CANCELLED` | 申请或契约取消 |

建议后续将“接单申请”和“任务契约”分表处理：

```text
ApplicationStatus = APPLIED / ACCEPTED / REJECTED / CANCELLED
ContractStatus = IN_PROGRESS / WAIT_CONFIRM / COMPLETED / CANCELLED / DISPUTED / RULED
```

### 5.3 申诉状态到小法庭状态

| 原申诉状态 | 小法庭状态 | 说明 |
| --- | --- | --- |
| `PENDING` | `FILED` | 已立案，等待处理 |
| `PROCESSING` | `EVIDENCE_COLLECTING` 或 `VOTING` | 证据收集或陪审团投票中 |
| `RESOLVED` | `RULED` | 管理员已裁决 |
| `REJECTED` | `ARCHIVED` | 案件驳回或归档 |

建议完整小法庭状态：

```text
FILED
EVIDENCE_COLLECTING
AI_SUMMARIZED
VOTING
ADMIN_REVIEW
RULED
ARCHIVED
```

## 6. 阶段性改造路线

### 6.1 第 0 阶段：文档与接口契约对齐

目标是让前后端在动手前统一口径：

1. 确认业务术语：委托人、猎人、任务、赏金、接单申请、履约证据、小法庭；
2. 确认 API 前缀：`/api/tasks`、`/api/applications`、`/api/contracts`、`/api/court-cases`；
3. 确认可复用接口：登录注册、用户信息、文件上传、消息列表、后台审计；
4. 确认需要新增的数据对象：任务、申请、契约、证据、案件、投票、裁决、猎人档案。

本仓库新增的 4 份赏金布文档属于第 0 阶段产物。

### 6.2 第 1 阶段：最小任务平台

完成 A 档主链路，让系统从“二手商品交易”转成“校园任务接单”：

1. 用户注册登录与基础资料；
2. 任务发布、任务大厅、任务详情；
3. 猎人申请接单；
4. 委托人选择猎人；
5. 任务契约生成；
6. 猎人提交履约证据；
7. 委托人确认完成；
8. 双方评价；
9. 基础经验和信誉更新；
10. 关键节点消息通知。

这一阶段可先复用旧表结构做小步改造，但接口层应优先使用新路径，避免前端继续绑定商城语义。

### 6.3 第 2 阶段：小法庭与 AI 书记官

完成平台差异化能力：

1. 任务纠纷立案；
2. 双方陈述和证据提交；
3. AI 案件摘要、证据分析、幽默点评；
4. 陪审团投票与加权统计；
5. 管理员裁决；
6. 裁决后更新任务状态、信誉、经验和违规记录；
7. 判例归档。

AI 输出只作为辅助，不直接发布任务、不直接裁决案件、不直接修改信誉。

### 6.4 第 3 阶段：公会成长与运营玩法

在主链路稳定后增加趣味化与运营能力：

1. 猎人等级、称号、徽章；
2. 公会榜单；
3. 任务难度、推荐猎人等级；
4. AI 风险提醒；
5. 判例库引用；
6. 翻车复盘；
7. 运营配置、敏感词和高风险任务审核。

## 7. 优先级边界

### A 档：必须完成

| 能力 | 说明 | 建议归属 |
| --- | --- | --- |
| 登录注册 | 沿用现有 auth | `cm-auth` |
| 任务发布 | 商品发布改造 | `cm-product` |
| 任务大厅 | 商品列表改造 | `cm-product` |
| 任务详情 | 商品详情改造 | `cm-product` |
| 接单申请 | 订单创建前置化 | `cm-trade` |
| 选择猎人 | 生成契约 | `cm-trade` |
| 履约证据 | 发货逻辑改造为证据提交 | `cm-trade` + `cm-product` 文件上传 |
| 确认完成 | 收货完成改造 | `cm-trade` |
| 双方评价 | 原评价扩展 | `cm-trade` |
| 消息通知 | 沿用 message | `cm-notify` |

### B 档：核心亮点

| 能力 | 说明 | 建议归属 |
| --- | --- | --- |
| 小法庭立案 | 申诉升级为案件 | `cm-notify` 或后续独立 `cm-court` |
| 案件证据 | 与任务证据区分 | `cm-notify` |
| AI 案件摘要 | 书记官摘要 | 过渡期 `cm-product` 的 AI 能力，后续独立 |
| 陪审团投票 | 等级加权投票 | `cm-notify` + `cm-auth` |
| 管理员裁决 | 后台处理 | `cm-notify` |
| 判例归档 | 裁决沉淀 | `cm-notify` |

### C 档：锦上添花

| 能力 | 说明 |
| --- | --- |
| 猎人徽章 | 运营激励，依赖任务完成和评价数据 |
| 公会榜单 | 可先用聚合统计实现 |
| AI 判例引用 | 可先用内置规则模拟 |
| 任务风险提醒 | 发布和接单前提示风险 |
| 多种 AI 点评风格 | 演示增强，不影响主链路 |
| 公会税/积分 | 涉及结算规则，建议放后 |

## 8. 可复用、需改造、需新增清单

### 8.1 可复用

1. `ApiResponse<T>` 统一响应结构：`code/message/data`；
2. JWT 登录鉴权与 `Authorization: Bearer <token>`；
3. 网关统一入口与 CORS；
4. `/api/auth/login`、`/api/auth/register`、`/api/auth/me`；
5. `/api/users/{id}`、`/api/users/profile`；
6. `/api/files/upload` 与 `/uploads/**`；
7. `/api/messages`、`/api/messages/unread-count`、已读操作；
8. `/api/admin/audit-logs` 与内部审计思路；
9. 现有 Feign 内部调用和 `X-Internal-Token` 机制。

### 8.2 需改造

1. `ProductController/ProductService/ProductVO` 改为任务语义；
2. `ProductCategory/ProductCondition/ProductStatus` 改为任务分类、难度、状态；
3. `OrderController/OrderService/OrderVO` 拆分为接单申请和任务契约；
4. `Review` 从买家评价卖家扩展为双方评价；
5. `Appeal` 从普通申诉升级为小法庭案件；
6. 后台商品审核、订单查询改为任务审核、案件裁决和平台治理；
7. AI 商品描述、估价接口改造为任务拆解、赏金建议、风险提示、案件摘要。

### 8.3 需新增

1. 猎人档案 `HunterProfile`；
2. 校园认证 `CampusCertification`；
3. 任务申请 `TaskApplication`；
4. 任务契约 `TaskContract`；
5. 履约证据 `TaskEvidence`；
6. 任务状态日志 `TaskStatusLog`；
7. 小法庭案件 `CourtCase`；
8. 案件陈述 `CourtStatement`；
9. 案件证据 `CourtEvidence`；
10. 陪审团投票 `CourtVote`；
11. 裁决记录 `CourtRuling`；
12. 判例归档 `CourtPrecedent`；
13. AI 输出记录 `AiOutputLog`；
14. 信誉变化日志 `CreditLog`；
15. 徽章与用户徽章关系表。

## 9. 前后端协作原则

1. 前端从新业务路径开始接入，不再新增依赖 `/api/products` 和 `/api/orders` 的页面逻辑；
2. 后端过渡期可以在旧模块中实现新接口，但返回字段必须使用新业务语义；
3. 所有接口继续使用统一响应结构；
4. 所有写操作默认需要登录；
5. 后台接口统一走 `/api/admin/**` 并校验管理员角色；
6. 内部接口仍使用 `/internal/**`，不得暴露给前端；
7. AI 相关接口必须允许失败降级，失败时用户仍可手工发布任务或处理案件；
8. 小法庭投票和 AI 输出只作为参考，最终裁决由管理员确认。
