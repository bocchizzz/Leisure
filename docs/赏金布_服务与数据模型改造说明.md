# 赏金布服务与数据模型改造说明

版本：v0.1  
日期：2026-06-30  
适用对象：后端开发、数据库设计、架构设计、测试

## 1. 改造目标

当前项目是“校园二手商城”微服务，核心数据围绕商品、订单、评价、申诉展开。赏金布的目标是把它升级为“校园任务悬赏与接单平台”，核心数据应围绕任务、申请、契约、履约证据、评价、小法庭、猎人成长展开。

本说明给出后端服务职责和数据模型的过渡方案，原则是：

1. 不要求第一阶段物理重命名微服务；
2. 优先复用已有登录、网关、统一响应、文件上传、消息、审计能力；
3. 新业务接口使用赏金布语义；
4. 数据模型先保证主链路，再扩展小法庭和公会成长；
5. 旧表可以作为迁移参考，但最终模型不应继续保留商品/买家/卖家语义。

## 2. 服务职责演进

### 2.1 `cm-gateway`

当前职责：

1. 统一入口；
2. 路由到 `cm-auth/cm-product/cm-trade/cm-notify`；
3. CORS；
4. JWT 是否存在的前置检查；
5. 禁止外部访问 `/internal/**`。

赏金布过渡职责：

1. 新增 `/api/tasks/**` 路由到 `cm-product`；
2. 新增 `/api/hunters/**`、`/api/certifications/**` 路由到 `cm-auth`；
3. 新增 `/api/applications/**`、`/api/contracts/**`、`/api/evidences/**` 路由到 `cm-trade`；
4. 新增 `/api/court-cases/**`、`/api/admin/court-cases/**` 路由到 `cm-notify`；
5. 新增 `/api/ai/tasks/**` 和 `/api/ai/court/**` 路由，过渡期可仍到 `cm-product`。

改造建议：

| 优先级 | 改造项 | 说明 |
| --- | --- | --- |
| 高 | 新增路由 | 支持新前端路径 |
| 高 | 白名单调整 | 公开任务列表和详情，其他需要登录 |
| 中 | 管理端路径统一 | `/api/admin/**` 管理员权限 |
| 中 | AI 限流预留 | 避免演示时连续调用导致卡顿 |

### 2.2 `cm-auth`

当前职责：

1. 注册、登录、登出；
2. 当前用户信息；
3. 用户资料修改；
4. 用户角色；
5. 用户状态；
6. 信誉分；
7. 管理端封禁/解封；
8. 内部用户摘要接口。

赏金布过渡职责：

1. 用户基础身份继续保留；
2. 增加校园认证；
3. 增加猎人档案；
4. 增加经验值、等级、称号；
5. 增加信誉日志；
6. 增加徽章和用户徽章；
7. 为投票权重提供用户等级、信誉、历史投票采纳率。

建议新增接口：

| 路径 | 说明 |
| --- | --- |
| `/api/certifications` | 提交校园认证 |
| `/api/certifications/me` | 查看我的认证 |
| `/api/admin/certifications/{id}/review` | 管理员审核认证 |
| `/api/hunters/{userId}` | 猎人主页 |
| `/api/hunters/me` | 我的猎人档案 |
| `/api/hunters/leaderboard` | 榜单 |
| `/internal/users/{id}/hunter-profile` | 服务间查询猎人档案 |
| `/internal/users/{id}/credit` | 服务间更新信誉或查询信誉 |

### 2.3 `cm-product`

当前职责：

1. 商品发布、编辑、删除、重上架；
2. 商品列表、详情、推荐；
3. 商品分类、成色、状态；
4. 收藏、浏览历史；
5. 文件上传；
6. 商品 AI 能力；
7. 内部商品预占/确认/取消。

赏金布过渡职责：

1. 任务发布、编辑、取消；
2. 任务大厅、任务搜索、任务筛选、推荐；
3. 任务分类、难度、状态；
4. 任务收藏、任务浏览历史；
5. 任务封面和附件上传；
6. AI 任务拆解、赏金建议、风险提示；
7. 为接单服务提供任务摘要和状态更新内部接口。

保留能力：

| 原能力 | 复用方式 |
| --- | --- |
| 列表分页 | 任务大厅分页 |
| 关键词搜索 | 任务搜索 |
| 分类筛选 | 任务分类筛选 |
| 价格筛选 | 赏金筛选 |
| 文件上传 | 任务封面、履约证据、小法庭证据 |
| 浏览历史 | 任务浏览记录 |
| 收藏 | 任务收藏 |
| 推荐 | 任务推荐 |

需要移除或弱化：

1. 商品成色；
2. 商品预占 Saga 语义；
3. 卖家/买家语义；
4. 售出语义；
5. 商品估价文案。

内部接口建议：

| 路径 | 说明 |
| --- | --- |
| `POST /internal/tasks/briefs` | 批量查询任务摘要 |
| `GET /internal/tasks/{id}/status` | 查询任务状态 |
| `PUT /internal/tasks/{id}/assign` | 标记任务已分配猎人 |
| `PUT /internal/tasks/{id}/wait-confirm` | 标记任务待确认 |
| `PUT /internal/tasks/{id}/complete` | 标记任务已完成 |
| `PUT /internal/tasks/{id}/dispute` | 标记任务争议中 |
| `PUT /internal/tasks/{id}/cancel` | 标记任务取消 |

### 2.4 `cm-trade`

当前职责：

1. 创建订单；
2. 确认订单；
3. 发货；
4. 完成订单；
5. 取消订单；
6. 买家/卖家订单列表；
7. 评价；
8. 调用商品服务进行预占、确认、取消、售出；
9. 调用通知服务发送消息。

赏金布过渡职责：

1. 猎人提交接单申请；
2. 委托人查看申请并选择猎人；
3. 生成任务契约；
4. 任务执行状态流转；
5. 猎人提交履约证据；
6. 委托人确认完成；
7. 取消契约；
8. 双方评价；
9. 完成后通知 `cm-auth` 更新经验和信誉；
10. 发生争议时通知 `cm-notify` 或由前端发起小法庭。

关键变化：

| 旧订单操作 | 新业务操作 |
| --- | --- |
| 创建订单 | 提交接单申请 |
| 卖家确认订单 | 委托人选择猎人 |
| 发货 | 猎人提交履约证据 |
| 买家确认完成 | 委托人确认完成 |
| 取消订单 | 取消申请或契约 |
| 买家评价卖家 | 双方评价 |

建议拆分对象：

1. `TaskApplication`：申请接单阶段；
2. `TaskContract`：委托人选中猎人后的任务契约；
3. `TaskEvidence`：猎人提交的履约证据；
4. `TaskReview`：双方评价。

### 2.5 `cm-notify`

当前职责：

1. 站内消息；
2. 未读数量；
3. 申诉；
4. 后台申诉处理；
5. 审计日志；
6. 后台看板。

赏金布过渡职责：

1. 保留站内消息；
2. 扩展消息类型；
3. 将申诉升级为小法庭案件；
4. 存储双方陈述、案件证据、AI 摘要、AI 点评；
5. 陪审团投票；
6. 管理员裁决；
7. 判例归档；
8. 裁决后调用 `cm-trade` 更新契约和任务状态；
9. 裁决后调用 `cm-auth` 更新信誉和等级相关数据；
10. 后台看板增加任务数、完成率、纠纷率、AI 调用次数等指标。

是否独立 `cm-court`：

| 方案 | 优点 | 缺点 | 建议 |
| --- | --- | --- | --- |
| 小法庭先放 `cm-notify` | 改造少，能快速完成课程项目 | `cm-notify` 职责变重 | 第一阶段推荐 |
| 新增 `cm-court` | 边界清晰，后续扩展好 | 工作量大，需要新增服务、路由、配置 | 主链路稳定后再做 |

### 2.6 `cm-common`

当前职责：

1. `ApiResponse<T>`；
2. 统一异常；
3. JWT 工具；
4. 安全过滤器；
5. Feign 拦截器和错误处理；
6. 请求 ID。

赏金布过渡职责：

1. 保持统一响应；
2. 保持 JWT 和内部 token；
3. 可增加公共枚举或常量，但避免把业务实体塞进 common；
4. 可增加统一的分页响应适配工具；
5. 可增加敏感操作日志辅助。

## 3. 数据库演进方案

### 3.1 当前数据库

当前拆分：

| 数据库 | 主要表 | 当前语义 |
| --- | --- | --- |
| `campus_user` | `sys_user`、`sys_user_role` | 用户、角色 |
| `campus_product` | `biz_product`、`biz_favorite`、`biz_browse_history` | 商品、收藏、浏览历史 |
| `campus_trade` | `biz_order`、`biz_review` | 订单、评价 |
| `campus_notify` | `biz_message`、`biz_appeal`、`sys_audit_log` | 消息、申诉、审计 |

### 3.2 目标数据库边界

建议仍保持四个数据库边界：

| 数据库 | 新职责 |
| --- | --- |
| `campus_user` | 用户、校园认证、猎人档案、等级、徽章、信誉日志 |
| `campus_product` | 任务、任务分类、任务收藏、任务浏览历史、任务状态日志 |
| `campus_trade` | 接单申请、任务契约、履约证据、双方评价 |
| `campus_notify` | 消息、小法庭案件、陈述、案件证据、投票、裁决、判例、审计 |

后续如果课程要求服务边界更清晰，可将 `campus_product` 改名为 `campus_task`，将 `campus_notify` 中的小法庭拆出 `campus_court`。第一阶段不强制。

## 6. 接单与履约模型

### 6.1 接单申请 `task_application`

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT` | 主键 |
| `task_id` | `BIGINT` | 任务 ID |
| `hunter_id` | `BIGINT` | 猎人 ID |
| `apply_message` | `TEXT` | 申请说明 |
| `expected_finish_time` | `DATETIME` | 预计完成时间 |
| `status` | `VARCHAR(20)` | `APPLIED/ACCEPTED/REJECTED/CANCELLED/EXPIRED` |
| `created_at` | `DATETIME` | 申请时间 |
| `updated_at` | `DATETIME` | 更新时间 |

约束建议：

1. 同一任务同一猎人只能有一条有效申请；
2. 委托人不能申请自己的任务；
3. 任务状态必须是 `PUBLISHED` 或 `APPLYING` 才能申请；
4. 接受某个申请后，同任务其他申请自动变为 `REJECTED` 或保持未选中状态。

### 6.2 任务契约 `task_contract`

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT` | 主键 |
| `contract_no` | `VARCHAR(50)` | 契约编号 |
| `task_id` | `BIGINT` | 任务 ID |
| `application_id` | `BIGINT` | 来源申请 |
| `publisher_id` | `BIGINT` | 委托人 |
| `hunter_id` | `BIGINT` | 猎人 |
| `bounty_amount` | `DECIMAL(10,2)` | 赏金快照 |
| `task_title` | `VARCHAR(200)` | 任务标题快照 |
| `completion_standard` | `TEXT` | 完成标准快照 |
| `evidence_requirement` | `TEXT` | 证据要求快照 |
| `status` | `VARCHAR(30)` | 契约状态 |
| `reviewed_by_publisher` | `TINYINT(1)` | 委托人是否已评价 |
| `reviewed_by_hunter` | `TINYINT(1)` | 猎人是否已评价 |
| `cancel_reason` | `VARCHAR(500)` | 取消原因 |
| `accepted_at` | `DATETIME` | 选中时间 |
| `submitted_at` | `DATETIME` | 提交完成时间 |
| `completed_at` | `DATETIME` | 完成时间 |
| `created_at/updated_at` | `DATETIME` | 时间 |

状态流转：

```text
IN_PROGRESS
-> WAIT_CONFIRM
-> COMPLETED
```

异常流转：

```text
IN_PROGRESS/WAIT_CONFIRM
-> DISPUTED
-> RULED
```

取消流转：

```text
IN_PROGRESS
-> CANCELLED
```

### 6.3 履约证据 `task_evidence`

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT` | 主键 |
| `contract_id` | `BIGINT` | 契约 ID |
| `task_id` | `BIGINT` | 任务 ID |
| `submitter_id` | `BIGINT` | 提交人 |
| `type` | `VARCHAR(20)` | `TEXT/IMAGE/FILE/LINK` |
| `file_url` | `VARCHAR(500)` | 文件地址 |
| `content` | `TEXT` | 说明 |
| `created_at` | `DATETIME` | 时间 |

规则：

1. 只有选中的猎人可以提交履约证据；
2. 契约必须处于 `IN_PROGRESS` 或 `WAIT_CONFIRM`；
3. 提交证据后可将契约状态推进到 `WAIT_CONFIRM`；
4. 证据不应被物理删除，后续纠纷可能引用。

### 6.4 双方评价 `task_review`

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT` | 主键 |
| `contract_id` | `BIGINT` | 契约 ID |
| `task_id` | `BIGINT` | 任务 ID |
| `reviewer_id` | `BIGINT` | 评价人 |
| `reviewee_id` | `BIGINT` | 被评价人 |
| `role` | `VARCHAR(30)` | `PUBLISHER_TO_HUNTER/HUNTER_TO_PUBLISHER` |
| `rating` | `TINYINT` | 1-5 分 |
| `tags` | `VARCHAR(500)` | 标签，逗号分隔或 JSON |
| `content` | `TEXT` | 内容 |
| `created_at` | `DATETIME` | 时间 |

约束：

1. 同一契约同一评价方向只能评价一次；
2. 契约必须是 `COMPLETED` 或裁决后允许评价的状态；
3. 评价成功后触发经验和信誉更新；
4. 评价内容进入内容安全检测。

### 7.5 陪审团投票 `court_vote`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT` | 主键 |
| `case_id` | `BIGINT` | 案件 ID |
| `voter_id` | `BIGINT` | 投票人 |
| `option` | `VARCHAR(50)` | 投票选项 |
| `weight` | `DECIMAL(5,2)` | 投票权重 |
| `reason` | `TEXT` | 理由 |
| `is_adopted` | `TINYINT(1)` | 是否与最终裁决一致 |
| `created_at` | `DATETIME` | 时间 |

投票权重建议：

```text
vote_weight = min(3.0, 1.0 + 0.25 * hunter_level + 0.3 * arbitration_accuracy)
```

其中：

```text
arbitration_accuracy = 历史投票被采纳次数 / 历史参与投票次数
```

第一版可简化：

```text
vote_weight = min(3.0, 1.0 + 0.25 * hunter_level)
```

| `reason` | `TEXT` | 裁决理由 |
| `created_at` | `DATETIME` | 时间 |

裁决后动作：

1. 更新 `court_case.status = RULED`；
2. 更新 `task_contract.status = RULED` 或根据结果进入 `COMPLETED/CANCELLED`；
3. 更新 `task.status`；
4. 写入双方 `credit_log`；
5. 更新猎人档案统计；
6. 将投票与裁决一致的 `court_vote.is_adopted` 标记为 true；
7. 可生成 `court_precedent`。

## 8. AI 输出模型

### 8.1 AI 能力边界

AI 书记官提供辅助，不直接执行业务决策：

| 场景 | AI 输出 | 是否直接生效 |
| --- | --- | --- |
| 发布任务 | 标题、分类、步骤、赏金建议、风险提示 | 否，用户确认后生效 |
| 接单前 | 风险提醒 | 否，猎人自行决定 |
| 小法庭 | 案件摘要、争议焦点、证据分析、点评 | 否，管理员裁决 |
| 判例引用 | 相似判例建议 | 否，仅参考 |

## 10. 跨服务数据流

### 10.1 发布任务

```text
前端 -> gateway -> cm-product
cm-product 保存 task
cm-product -> cm-notify/internal/messages 通知委托人
```

如启用任务审核：

```text
task.status = PENDING_REVIEW
管理员审核通过后 task.status = PUBLISHED
```

### 10.2 猎人申请接单

```text
前端 -> gateway -> cm-trade
cm-trade -> cm-product/internal/tasks/{id}/status 校验任务可申请
cm-trade -> cm-auth/internal/users/{id}/status 校验猎人状态
cm-trade 保存 task_application
cm-trade -> cm-notify/internal/messages 通知委托人
cm-trade -> cm-product/internal/tasks/{id}/application-count 增加申请数
```

### 10.3 委托人选择猎人

```text
前端 -> gateway -> cm-trade
cm-trade 校验申请属于该任务
cm-trade 校验当前用户是 publisher
cm-trade 将 application 标记 ACCEPTED
cm-trade 创建 task_contract
cm-trade -> cm-product/internal/tasks/{id}/assign 更新任务状态和 assigned_hunter_id
cm-trade -> cm-notify/internal/messages 通知猎人和其他申请人
```

### 10.4 猎人提交履约证据

```text
前端 -> /api/files/upload 上传文件
前端 -> /api/contracts/{id}/evidences 提交证据
cm-trade 保存 task_evidence
cm-trade 更新 contract.status = WAIT_CONFIRM
cm-trade -> cm-product/internal/tasks/{id}/wait-confirm
cm-trade -> cm-notify/internal/messages 通知委托人
```

### 10.5 委托人确认完成

```text
前端 -> /api/contracts/{id}/confirm-completion
cm-trade 更新 contract.status = COMPLETED
cm-trade -> cm-product/internal/tasks/{id}/complete
cm-trade -> cm-auth/internal/users/{hunterId}/xp 增加经验
cm-trade -> cm-notify/internal/messages 通知猎人
```

### 10.6 双方评价

```text
前端 -> /api/reviews
cm-trade 保存 task_review
cm-trade 更新 contract.reviewed_by_xxx
cm-trade -> cm-auth/internal/users/{revieweeId}/credit 更新信誉
cm-trade -> cm-auth/internal/hunters/{revieweeId}/stats 更新猎人统计
cm-trade -> cm-notify/internal/messages 通知被评价人
```

### 10.7 小法庭裁决

```text
前端 -> /api/court-cases 发起案件
cm-notify 保存 court_case
cm-notify -> cm-trade/internal/contracts/{id}/dispute
cm-notify -> cm-product/internal/tasks/{id}/dispute

管理员裁决：
前端 -> /api/admin/court-cases/{id}/rule
cm-notify 保存 court_ruling
cm-notify 更新 court_case.status = RULED
cm-notify -> cm-trade/internal/contracts/{id}/rule
cm-notify -> cm-product/internal/tasks/{id}/rule
cm-notify -> cm-auth/internal/users/{id}/credit 写信誉变化
cm-notify -> cm-notify/internal/messages 通知双方
```

## 11. 迁移落地顺序

推荐后端按以下顺序推进：

1. 网关新增新路径；
2. `cm-product` 新增任务接口，先复用旧商品表或新增任务表；
3. `cm-trade` 新增申请和契约模型；
4. `cm-trade` 完成履约证据和确认完成；
5. `cm-trade` 改造评价为双方评价；
6. `cm-auth` 增加猎人档案和基础经验更新；
7. `cm-notify` 扩展消息类型；
8. `cm-notify` 将申诉升级为小法庭案件；
9. 增加 AI 任务拆解和案件摘要；
10. 增加后台任务审核与案件裁决；
11. 增加榜单、徽章、判例库等扩展功能。

## 12. 最小可交付模型

如果开发时间有限，第一版只需保证以下表或等价数据结构：

1. `sys_user`；
2. `task`；
3. `task_application`；
4. `task_contract`；
5. `task_evidence`；
6. `task_review`；
7. `biz_message`；
8. `court_case`；
9. `court_vote`；
10. `court_ruling`。

第一版可暂缓：

1. 徽章；
2. 判例库；
3. AI 输出日志；
4. 精细化等级规则表；
5. 公会税/积分结算；
6. 独立 `cm-court` 服务；
7. 复杂反作弊投票规则。

## 13. 风险与注意事项

1. 不要继续在新接口中暴露 `product/seller/buyer/order` 字段名，否则前端会被旧语义锁死；
2. 不要把接单申请和任务契约混成一个对象，否则多个猎人申请、委托人选择会很难表达；
3. 履约证据和小法庭证据是两个不同业务对象，可复用上传能力但不要混表到不可区分；
4. 小法庭裁决必须写日志，后续信誉变化和任务状态都要可追溯；
5. AI 输出必须可失败，不能把主链路绑定到模型服务；
6. 管理员裁决要优先于投票结果，投票只做参考；
7. 内部接口不要通过网关公开；
8. 数据库跨服务不使用外键，服务层要负责一致性校验；
9. 课程项目可以先使用同步 Feign 调用，后续再考虑事件消息；
10. 若保留旧表过渡，接口层必须完成字段语义转换。
