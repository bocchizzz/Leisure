# **Campus Quest：校园猎人公会任务悬赏平台**

副标题：

> **一个集任务悬赏、猎人等级、小法庭仲裁、民意投票与 AI 毒舌书记官于一体的校园任务公会系统。**

这个设定比普通“AI 任务平台”有记忆点很多，而且依然符合课程要求：老师文档允许各组在统一框架下结合自身选题做个性化项目，但要求文档完整、架构清晰、分工明确、AI 与微服务解耦。 你们这套设定只要主链路不乱，完全可以做成一个很有趣的工程项目。

---

## 一、核心设计思路：主业务稳，外层设定疯

你们不要把项目做成纯玩梗系统。正确结构应该是：

> **底层是严肃的任务悬赏系统，上层是游戏化的猎人公会设定。**

也就是：

| 层级     | 内容                    | 作用      |
| ------ | --------------------- | ------- |
| 基础业务层  | 发布任务、接单、履约、评价、申诉      | 保证项目能跑通 |
| 公会机制层  | 猎人等级、声望、称号、徽章、任务难度    | 增加趣味和激励 |
| 小法庭层   | 纠纷提交、证据上传、民意投票、管理员裁决  | 增加差异化亮点 |
| AI 戏剧层 | AI 总结、AI 毒舌点评、AI 判例摘要 | 形成答辩记忆点 |

参考文档里也强调，要先保证主链路，再做 AI 扩展和联调；如果时间不够，要优先保留主业务链路和最小 AI 接口。 所以你们可以大胆设计，但开发时要按“先主线、后玩法”的顺序推进。

## 1. 项目名称

推荐几个名字：

1. **Campus Quest：校园猎人公会任务悬赏平台**

2. **猎人公会：AI 校园任务悬赏与仲裁系统**

3. **QuestHub：校园悬赏任务与小法庭平台**

4. **校园猎人榜：任务悬赏、等级成长与 AI 仲裁平台**

5. **Campus Court Quest：校园任务公会与民意小法庭**

我最推荐：

# **Campus Quest：校园猎人公会任务悬赏平台**

这个名字有任务、有校园、有世界观，不像普通课程项目。

---

## 2. 一句话介绍

> Campus Quest 面向高校学生的碎片化互助需求，构建一个支持任务发布、猎人接单、履约举证、等级成长、民意小法庭和 AI 犀利点评的校园任务悬赏平台。

---

## 3. 世界观设定

你们可以把用户分成几类：

| 角色    | 设定名             | 实际业务含义    |
| ----- | --------------- | --------- |
| 任务发布者 | 委托人             | 发布任务的人    |
| 接单者   | 猎人              | 接任务并完成的人  |
| 管理员   | 公会管理员 / 法庭庭长    | 审核任务、处理纠纷 |
| 投票用户  | 陪审团             | 参与纠纷投票    |
| AI    | AI 书记官 / 毒舌审判助理 | 总结证据、生成点评 |
| 高等级用户 | 高阶猎人 / 星级陪审员    | 投票权重更高    |

这样项目立刻从普通任务平台变成“校园公会系统”。

# 三、主业务闭环

你们最终主链路可以写成：

```text
用户注册登录
→ 发布悬赏任务
→ AI 拆解任务并建议赏金
→ 猎人申请接单
→ 委托人选择猎人
→ 猎人完成任务并提交履约证据
→ 委托人确认完成
→ 双方评价
→ 系统更新猎人经验、等级和信誉
→ 如有纠纷，进入校园小法庭
→ AI 总结证据 + 毒舌点评
→ 陪审团民意投票
→ 管理员最终裁决
→ 结果影响双方信誉和等级
```

这个闭环非常清楚，也符合启动会文档对“必须说清项目解决什么问题、主链路是什么、服务边界怎么切”的要求。

# 四、“小法庭”怎么设计最有意思

## 1. 小法庭定位

小法庭不是严肃司法系统，而是平台内部的纠纷处理机制。

可以叫：

> **校园小法庭 / 公会审判庭 / 猎人仲裁庭**

适用场景：

| 纠纷类型 | 示例              |
| ---- | --------------- |
| 履约争议 | 猎人说完成了，委托人说没完成  |
| 质量争议 | PPT 做了，但委托人觉得太丑 |
| 超时争议 | 猎人迟到，任务价值降低     |
| 报酬争议 | 委托人临时压价         |
| 证据争议 | 双方上传的证明互相矛盾     |
| 恶意任务 | 委托人发布不合理任务      |
| 恶意接单 | 猎人接单后消失         |

---

## 2. 小法庭流程

```text
发起纠纷
→ 双方提交证据
→ AI 书记官生成案件摘要
→ AI 犀利点评生成
→ 陪审团投票
→ 系统计算民意倾向
→ 管理员查看 AI 摘要、投票结果和证据
→ 管理员裁决
→ 更新任务状态、信誉分、猎人经验
→ 生成判例卡片
```

页面上可以做得很有戏剧感：

| 页面模块     | 展示内容                       |
| -------- | -------------------------- |
| 案件标题     | “快递代拿案：三件快递到底有没有送到？”       |
| 双方陈述     | 委托人说法、猎人说法                 |
| 证据墙      | 图片、文字说明、时间线                |
| AI 书记官摘要 | 客观总结事件经过                   |
| AI 毒舌点评  | 犀利但不人身攻击的吐槽                |
| 陪审团投票    | 支持委托人 / 支持猎人 / 证据不足 / 建议和解 |
| 公会裁决     | 管理员最终处理结果                  |
| 判例归档     | 形成“校园判例库”                  |

# 五、猎人等级体系

## 1. 等级名称

可以不要普通 Lv1、Lv2，做成更有世界观的称号。

| 等级  | 称号   | 经验范围      | 权益         |
| --- | ---- | --------- | ---------- |
| Lv0 | 见习猎人 | 0-99      | 可接普通任务     |
| Lv1 | 铁牌猎人 | 100-299   | 可参与小法庭投票   |
| Lv2 | 铜牌猎人 | 300-699   | 投票权重略高     |
| Lv3 | 银牌猎人 | 700-1499  | 可接高赏金任务    |
| Lv4 | 金牌猎人 | 1500-2999 | 小法庭投票权重提高  |
| Lv5 | 星徽猎人 | 3000-5999 | 可被系统优先推荐   |
| Lv6 | 传奇猎人 | 6000+     | 可担任“荣誉陪审员” |

也可以再中二一点：

| 等级  | 称号      |
| --- | ------- |
| Lv0 | 菜鸟猎人    |
| Lv1 | 跑腿新星    |
| Lv2 | 校园游侠    |
| Lv3 | 银月猎手    |
| Lv4 | 金章猎人    |
| Lv5 | 星辉仲裁者   |
| Lv6 | 传说级校园猎人 |

---

## 2. 经验值计算公式

你们可以在详细设计里写一个简单公式，显得很工程化。

```text
XP = 10 × 完成任务数
   + 15 × 好评数
   + 20 × 被采纳的仲裁投票数
   - 10 × 超时次数
   - 30 × 违规次数
   - 50 × 恶意纠纷次数
```

推导逻辑是：

完成任务体现贡献，好评体现质量，仲裁投票被采纳体现判断能力；超时、违规、恶意纠纷会破坏平台秩序，所以要扣分。

---

## 3. 猎人信誉分

经验值决定等级，信誉分决定可信度。两者不要混在一起。

```text
信誉分 = 100
      + 2 × 好评数
      + 1 × 准时完成数
      - 5 × 差评数
      - 8 × 超时数
      - 15 × 败诉纠纷数
      - 30 × 严重违规数
```

经验值可以慢慢涨，信誉分可以涨也可以跌。这样更合理。

# 六、民意投票 + 高等级权重

你们这个“猎人等级越高权重越高，但可以民意投票”的设定很好。关键是要防止变成“高等级垄断裁判”。所以我建议采用：

> **等级加权投票 + 普通民意修正 + 管理员最终裁决**

AI 和投票都不直接决定结果，只给管理员参考。这样既好玩，又安全。

---

## 1. 投票选项

每个案件投票可以有 4 个选项：

1. 支持委托人

2. 支持猎人

3. 证据不足

4. 建议和解

---

## 2. 投票权重公式

每个投票用户的权重：

```text
vote_weight = min(3.0, 1.0 + 0.25 × 猎人等级 + 0.3 × 仲裁可信系数)
```

其中：

```text
仲裁可信系数 = 历史投票被最终裁决采纳次数 / 历史参与投票次数
```

例如：

| 用户   | 猎人等级 | 仲裁可信系数 | 权重   |
| ---- | ---- | ------ | ---- |
| 普通用户 | 0    | 0      | 1.00 |
| 铁牌猎人 | 1    | 0.5    | 1.40 |
| 银牌猎人 | 3    | 0.6    | 1.93 |
| 传奇猎人 | 6    | 0.8    | 2.74 |
| 极限上限 | 任意   | 任意     | 3.00 |

设置 `min(3.0, ...)` 的原因是：高等级应有影响力，但不能一票顶十票，否则民意投票会失去意义。

---

## 3. 最终民意倾向计算

```text
加权支持率 = 某选项加权票数 / 所有加权票数
```

例如：

```text
支持猎人票数 = Σ 支持猎人的 vote_weight
总票数 = Σ 所有 vote_weight
支持猎人率 = 支持猎人票数 / 总票数
```

管理员页面可以展示：

| 指标    | 示例  |
| ----- | --- |
| 支持委托人 | 31% |
| 支持猎人  | 52% |
| 证据不足  | 12% |
| 建议和解  | 5%  |

然后系统给出：

> 当前民意倾向：轻微偏向猎人，但证据仍需管理员复核。

---

## 4. 防刷票机制

可以设计几个简单规则：

| 规则               | 作用       |
| ---------------- | -------- |
| 只有 Lv1 以上可参与正式投票 | 防止新号刷票   |
| 同一案件投票后不可修改      | 保证投票稳定   |
| 与案件双方存在交易关系的用户降权 | 防止熟人团    |
| 短时间集中投票触发异常提示    | 防止拉票     |
| 管理员裁决优先于投票结果     | 避免群众审判失控 |

这个部分放到非功能需求里的“安全性与治理机制”非常好写。

# 七、AI 书记官 + AI 毒舌点评

这个是你们最容易出彩的部分。

但要注意：  
**AI 可以犀利，但不能做人身攻击。**  
建议定位为：

> **AI 毒舌书记官：只吐槽行为和证据，不攻击身份、外貌、人格、地域、性别等。**

---

## 1. AI 输出分三层

| 模块       | 风格     | 用途             |
| -------- | ------ | -------------- |
| AI 书记官摘要 | 客观、中立  | 给管理员和投票者快速理解案件 |
| AI 证据分析  | 理性、条理化 | 判断双方证据强弱       |
| AI 犀利点评  | 毒舌、幽默  | 增加系统记忆点和演示效果   |

---

## 2. AI 书记官摘要示例

案件：快递代拿争议案。

```text
AI 书记官摘要：

本案中，委托人发布“代拿三件快递并送至 7 号楼”的任务，赏金 8 元。
猎人于 16:20 接单，16:58 上传完成照片，照片显示快递放置在 7 号楼大厅桌面。
委托人于 17:10 提出争议，称未收到快递。
当前证据显示，猎人提供了送达照片，但照片中无法明确显示快递单号；委托人未提供快递丢失的进一步证明。
因此，本案核心争议为：猎人上传的照片能否证明任务已完成。
```

---

## 3. AI 证据分析示例

```text
AI 证据分析：

1. 猎人证据：
   - 有送达照片，时间接近任务完成时间。
   - 但照片未包含完整快递单号，证明力中等。

2. 委托人证据：
   - 提交了“未收到”的文字说明。
   - 暂无聊天记录、监控截图或快递状态截图，证明力较弱。

3. 初步建议：
   - 若无更多证据，建议判定猎人基本履约，但扣除部分信誉分作为证据不完整提醒。
```

---

## 4. AI 毒舌点评示例

你们可以设计得非常有记忆点：

```text
AI 毒舌点评：

这案子的证据强度，大概相当于食堂阿姨说“我真的给你多打了一块肉”。
猎人拍了照片，但没拍清单号；委托人说没收到，但也没拿出更多证明。
一个像把作业写了但忘记署名，一个像说自己复习了但试卷空着。
建议双方下次别只靠嘴硬，证据拍清楚，比吵三百句都有用。
```

或者：

```text
AI 犀利点评：

本案目前呈现出一种经典校园纠纷美学：
一方说“我送了”，一方说“我没见到”，证据却像考试前的复习计划一样，看起来存在，但不太扎实。
建议猎人以后拍照带上快递单号，委托人也别只提交一句“我没收到”就想逆风翻盘。
```

这个会非常适合演示。

---

## 5. AI 毒舌风格可切换

页面上可以做一个“点评风格”选择：

| 风格     | 示例                  |
| ------ | ------------------- |
| 客观书记官  | 严肃总结                |
| 毒舌模式   | 犀利吐槽                |
| 老师批改模式 | “这份证据不够规范”          |
| 法庭审判模式 | “本庭认为……”            |
| 校园广播模式 | “各位同学请注意……”         |
| 阴阳怪气模式 | “你们俩都挺有道理，就是证据都不太行” |

注意：阴阳怪气模式只用于演示，文档中写成“幽默化反馈模式”更正式。

# 八、更多有意思的玩法设定

下面这些你们可以挑 5-8 个做，不要全做。

---

## 1. 任务难度评级

每个任务由 AI 或发布者设定难度：

| 难度  | 名称   | 示例          |
| --- | ---- | ----------- |
| F   | 顺手任务 | 顺路带饭        |
| E   | 跑腿任务 | 取快递         |
| D   | 技能任务 | 修电脑         |
| C   | 学习任务 | 讲题、整理资料     |
| B   | 创作任务 | PPT 美化、海报设计 |
| A   | 高难任务 | 活动策划、视频剪辑   |
| S   | 传说任务 | 紧急救场、复杂项目协助 |

页面可以显示：

> 当前任务等级：B 级创作任务  
> 推荐猎人等级：Lv3 及以上

---

## 2. 任务赏金 + 公会税

可以设计一个虚拟积分，不涉及真实支付。

```text
任务赏金 = 发布者设置积分
平台公会税 = 任务赏金 × 5%
猎人实际获得 = 任务赏金 - 公会税
```

公会税可以用于：

| 用途    | 说明      |
| ----- | ------- |
| 公会基金  | 奖励优质猎人  |
| 小法庭运营 | 仲裁投票奖励  |
| 新手补贴  | 鼓励新猎人接单 |

这会让系统更像一个完整生态。

---

## 3. 猎人徽章

| 徽章     | 获得条件               |
| ------ | ------------------ |
| 快递侠    | 完成 10 次跑腿任务        |
| PPT 圣手 | 完成 5 次设计任务且好评率 90% |
| 守时怪物   | 连续 20 次任务未超时       |
| 纠纷绝缘体  | 完成 30 次任务无争议       |
| 法庭常胜客  | 参与仲裁投票 10 次且采纳率高   |
| 救火队长   | 完成 3 次紧急任务         |
| 校园万事屋  | 完成 5 种不同类型任务       |

---

## 4. 黑榜与红榜

做一个“公会榜单”页面：

| 榜单    | 内容          |
| ----- | ----------- |
| 猎人荣耀榜 | 等级最高、完成最多   |
| 守时榜   | 准时率最高       |
| 好评榜   | 好评率最高       |
| 仲裁之眼榜 | 投票最准确       |
| 翻车榜   | 争议最多，展示时匿名化 |
| 鸽王警告榜 | 接单后取消最多     |

“翻车榜”和“鸽王榜”要注意隐私，建议只显示匿名或模拟数据。

---

## 5. 判例库

小法庭处理完的案件可以形成“校园判例库”。

例如：

| 判例名       | 结论                  |
| --------- | ------------------- |
| 快递照片不清案   | 送达照片未包含关键信息，证据力降低   |
| PPT 审美争议案 | 审美主观差异不构成严重违约       |
| 代拿奶茶迟到案   | 超过约定时间导致饮品变质，猎人部分担责 |
| 讲题质量争议案   | 若任务未约定讲解深度，不宜全额扣赏金  |

这个特别适合写测试报告和课程设计报告，因为它能体现系统治理能力。

---

## 6. AI 判例引用

AI 在新案件中可以引用历史判例：

```text
AI 参考判例：

本案与“快递照片不清案”相似，二者均存在履约照片无法证明关键事实的问题。
根据历史判例，若猎人能证明已到达指定地点但无法证明物品准确交付，可判定为部分履约。
```

技术上你们可以不做真正 RAG，先用几条内置规则模拟即可。参考文档中也提到 AI 可以作为独立能力模块，业务层不直接耦合模型厂商，最小能力可先做对话、文案、审核、摘要等。

---

## 7. 猎人宣言

用户主页可以生成“猎人宣言”：

```text
我是 Lv4 金牌猎人，擅长跑腿、PPT 美化和临时救场。
我的原则是：不鸽单、不嘴硬、不让委托人凌晨还在等消息。
```

AI 可以一键生成。

## 8. 任务遗言 / 翻车复盘

任务失败后，系统生成“翻车复盘”：

```text
本次任务翻车原因：
1. 委托人没有写清交付标准。
2. 猎人没有及时反馈进度。
3. 双方都以为对方懂了，但其实谁也没懂。

AI 建议：
下次请把任务要求写得像考试题一样明确，而不是像朋友圈文案一样朦胧。
```

这个功能非常有趣，也能体现 AI 总结能力。

## 10. “别接！这是坑！”AI 风险提醒

AI 可以在猎人接单前提醒：

```text
AI 风险提醒：

该任务描述存在 3 个风险：
1. 交付标准不明确：“做得好看一点”无法量化。
2. 截止时间较紧，距离当前仅剩 40 分钟。
3. 委托人历史修改需求次数较多。

建议接单前要求委托人补充：具体格式、参考样例、验收标准。
```

这个比普通风控更有趣，而且非常实用。

# 十五、最终建议版本

你们现在可以把项目正式定为：

# **Campus Quest：校园猎人公会任务悬赏平台**

核心创新点写 5 个：

1. **任务悬赏机制**：把校园碎片化互助需求转化为可发布、可接单、可履约的任务。

2. **猎人等级体系**：根据完成任务、好评、守时率、仲裁表现动态计算等级与权重。

3. **校园小法庭**：任务纠纷进入证据提交、AI 总结、民意投票、管理员裁决流程。

4. **民意加权投票**：高等级猎人拥有更高投票权重，但设置上限，保留大众民意。

5. **AI 毒舌书记官**：AI 负责案件摘要、证据分析和幽默犀利点评，提升系统辨识度。

这套设定的优势是：  
**好玩，但不是乱玩；有梗，但有业务闭环；能演示，但也能写进正规软件工程文档。**
