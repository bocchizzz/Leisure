## 4. 自动化验收脚本

在 `frontend` 目录运行：

```bash
npm run test:e2e
```

当前 Playwright 自动启动 `npm run dev:mock -- --port 5177 --strictPort`，使用 Chromium 跑以下场景：

| 场景 | 覆盖点 |
| --- | --- |
| 公开页面与登录拦截 | 游客可进任务大厅，访问 `/my-tasks` 跳登录并携带 redirect |
| 登录表单 | `client01 / 123456` 可通过 UI 登录并写入 token |
| 认证/后台权限 | `newbie` 发布任务跳认证；普通用户访问后台被打回任务大厅 |
| 管理员审核 | `admin` 进入后台并通过待审核任务 |
| 任务与契约 | 委托人选择猎人生成契约；猎人提交证据/提交完成；委托人确认完成 |
| 小法庭 | 陪审员投票或识别已投票状态；管理员打开案件裁决抽屉 |

说明：当前 mock 是浏览器内存态，不是独立后端服务；跨浏览器上下文不会共享新生成的 mock 数据。因此自动化中“新建契约”和“履约/确认”分别使用可稳定复现的数据验证。真实后端接入后，可把同一条端到端流转升级为跨账号共享状态验收。

## 5. 手工验收脚本

### 5.1 主链路

1. 启用 mock，登录 `client01 / 123456`。
2. 进入 `/tasks/publish` 发布一条合规任务，确认进入后台待审。
3. 登录 `admin / 123456`，进入 `/admin/tasks`，查看内容安全状态并通过任务。
4. 登录 `hunter01 / 123456`，进入任务大厅，使用分类、难度、赏金区间和 AI 搜索筛选任务。
5. 打开任务详情并申请接单。
6. 登录 `client01`，进入 `/tasks/:id/applications` 选择猎人。
7. 登录 `hunter01`，进入 `/my-tasks` 的承接契约，提交履约证据并提交完成。
8. 登录 `client01`，确认完成。
9. 双方分别进入契约详情提交评价。
10. 查看 `/hunters/3`，确认评价、徽章、信誉日志变化。
11. 进入任务大厅点击“浏览历史”，确认最近打开的任务出现。

### 5.2 小法庭

1. 使用 `client01` 或 `hunter01` 打开争议契约，发起小法庭。
2. 双方在案件详情补充陈述和证据。
3. 点击 AI 摘要和书记官点评。
4. 登录 `jury01` 或 `hunter02`，打开案件详情投票。
5. 登录 `admin`，进入 `/admin/court-cases` 查看案件详情并裁决。
6. 勾选归档后检查 `/precedents` 是否出现判例。
7. 检查双方消息中心和猎人档案信誉日志。

### 5.3 后台治理

1. 登录 `admin`。
2. 检查 `/admin/dashboard` 指标。
3. 检查 `/admin/users` 封禁/解封用户。
4. 检查 `/admin/certifications` 通过/驳回认证。
5. 检查 `/admin/tasks` 任务安全状态、通过/驳回。
6. 检查 `/admin/ops-config` 修改审核阈值、关键词、上传限制、陪审团资格。
7. 检查 `/admin/audit-logs` 是否记录操作。

### 5.4 权限与异常

| 场景 | 预期 |
| --- | --- |
| 未登录访问 `/my-tasks` | 跳转登录 |
| `newbie` 发布任务 | 跳转校园认证 |
| `newbie` 申请任务 | 弹出认证门禁 |
| 普通用户访问 `/admin/dashboard` | 被拦截回前台 |
| 非契约双方访问契约详情 | mock 返回权限错误 |
| 重复申请同一任务 | 提示已申请过 |
| 重复评价同一契约 | 提示已评价过 |
| 案件双方尝试投票 | 提示案件双方不能投票 |
| 上传超大或不匹配类型文件 | 前端直接提示并阻止上传 |

## 6. 后端仍需兜底

前端 mock 只用于开发调试，以下规则必须由后端权威实现：

- JWT 鉴权、角色权限、校园认证状态、封禁状态。
- 任务、申请、契约、小法庭的合法状态机。
- 申请唯一性、重复评价、重复投票、任务双方权限。
- 文件真实存储、病毒扫描、隐私/敏感内容检测。
- 内容安全模型、AI 输出安全过滤、违规处置。
- 信誉分、经验、等级、徽章、审计日志、消息通知的权威计算与落库。
- 管理员裁决后的赏金释放、信誉变更、判例归档副作用。
