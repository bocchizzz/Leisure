下面这份可以当成你的“前端无后端 Mock 验收手册”。先说结论：你现在重点不是只看页面能不能打开，而是要按身份切换，把“能做什么、不能做什么、状态有没有变化、消息/审计/信誉有没有联动”全部走一遍。

**启动方式**

前端无后端完整验收时，在 `frontend` 目录启动 mock：

```powershell
npm run dev:mock
```

打开终端实际打印的地址。你现在机器上 `5173` 可能已有旧服务，所以可能是：

```text
http://127.0.0.1:5174/
```

后端联调时使用：

```powershell
npm run dev
```

`npm run dev` 不启用本地公开兜底，页面会严格展示后端 `/tasks`、`/tasks/recommendations`、`/hunters/leaderboard` 返回的数据。公开接口报错或返回空数组时，页面会显示错误/空态，方便定位后端问题。

注意：mock 是浏览器内存数据。刷新页面、重启 dev server、HMR 大更新，都可能让新增任务/申请/裁决回到初始状态。做完整链路时尽量在同一轮页面会话里连续测。

|---|---|---|---|---|

| `client01` | `123456` | 委托人，已认证 | 发布任务、查看自己任务申请、选择猎人、确认完成、评价猎人、作为小法庭当事人陈述/举证 | 主业务委托人链路 |
| `hunter01` | `123456` | 猎人，已认证 | 申请任务、提交履约证据、提交完成、评价委托人、作为争议案件当事人 | 猎人履约链路 |
| `hunter02` | `123456` | 竞争猎人/陪审员，已认证 | 可申请任务、也可作为陪审员投票，但不能投自己参与的案件 | 竞争申请/陪审员权限 |

**推荐主链路调试**

按这个顺序测，最容易发现前端逻辑问题：

1. 登录 `client01`。
2. 进入 `/tasks/publish` 发布一条合规任务，比如“帮忙取快递送到宿舍楼下”。
3. 确认发布后任务进入待审核状态。
4. 切换登录 `admin`，进入 `/admin/tasks`。
5. 查看任务的“内容安全”状态，点击通过。
6. 切换登录 `hunter01`。
7. 回到任务大厅 `/`，测试分类、难度、赏金区间、AI 搜索。
8. 打开刚发布的任务详情，提交申请。
9. 切换回 `client01`。
10. 进入 `/tasks/:id/applications`，选择 `hunter01`。
11. 确认生成契约。
12. 切换 `hunter01`，进入 `/my-tasks`，找到承接契约。
13. 进入 `/contracts/:id`，上传/填写履约证据。
14. 点击提交完成。
15. 切换 `client01`，进入同一契约，确认完成。
16. 双方分别提交评价。
17. 打开 `/hunters/3`，检查评价、信誉变化记录是否更新。
18. 打开 `/messages`，检查申请、契约、证据、完成、评价消息。

**可直接用的种子数据**

不用从零造数据时，可以直接测这些：

| 数据 | 状态 | 推荐用途 |
|---|---|---|
| 任务 `101` | 已发布，委托人 `client01`，已有 `hunter02` 申请 | 测委托人查看申请/选择猎人 |

| 任务 `103` | 待审核，安全状态 `REVIEW` | 测后台任务安全复核 |
| 任务 `109` | 待审核，安全状态 `BLOCKED` | 测违规内容驳回 |
| 契约 `301` | 进行中，双方 `client01` / `hunter01` | 测猎人提交证据、提交完成 |
| 契约 `302` | 待确认 | 测委托人确认完成 |
| 契约 `303` | 已完成，猎人已评价，委托人未评价 | 测 `client01` 评价猎人 |
| 契约 `304` | 已争议 | 测小法庭案件 |
| 案件 `401` | 投票中 | 测小法庭详情、AI 摘要、举证、裁决 |

**小法庭调试**

现成案件：

1. 登录 `client01`，打开 `/court/401`。
2. 检查自己是原告，可补充陈述、提交证据。
3. 点击 AI 摘要、AI 点评。
4. 尝试投票，预期：案件双方不能投票。
5. 登录 `hunter01`，打开 `/court/401`。
6. 检查自己是被告，可补充陈述、提交证据。
7. 登录 `jury01` 或 `hunter02` 打开 `/court/401`。
8. 注意：初始 mock 里 `jury01` 和 `hunter02` 已经有投票记录，可用来测“已投票/重复投票”提示。
9. 登录 `admin`，进入 `/admin/court-cases`。
10. 打开案件详情，填写裁决、赏金释放比例、信誉变化。
11. 勾选归档为判例。
12. 去 `/precedents` 检查判例是否出现。
13. 去 `/admin/audit-logs` 检查裁决审计日志。
14. 去 `/messages` 检查双方是否收到裁决消息。

如果要测“成功投票”，建议先通过新流程发起一个新的纠纷案件，再用 `jury01` 投票；现成 `401` 更适合测重复投票和当事人禁止投票。

**权限异常必须测**

| 场景 | 操作 | 预期 |
|---|---|---|
| 未登录访问 `/my-tasks` | 退出登录后打开 | 跳转登录 |
| 未登录访问 `/court` | 退出登录后打开 | 跳转登录 |
| `newbie` 访问 `/tasks/publish` | 登录 `newbie` 打开 | 跳转校园认证 |
| `newbie` 申请任务 | 打开已发布任务申请 | 提示需要认证 |
| `newbie` 投票 | 打开小法庭投票 | 提示需要认证/资格不足 |
| 普通用户访问 `/admin/dashboard` | `client01` 打开 | 拦截回前台 |
| 猎人申请自己发布的任务 | `hunter02` 申请任务 `108` | 提示不能申请自己发布的任务 |
| 重复申请 | 同一猎人重复申请同一任务 | 提示已申请过 |
| 非委托人查看申请列表 | `hunter01` 打开 `/tasks/101/applications` | 权限错误 |
| 非契约双方看契约 | 无关账号打开 `/contracts/301` | 权限错误 |
| 委托人提交履约证据 | `client01` 在契约提交证据 | 只有猎人可提交 |
| 猎人确认完成 | `hunter01` 确认完成 | 只有委托人可确认 |
| 重复评价 | 已评价一方再次评价 | 提示已评价过 |
| 案件双方投票 | `client01` 或 `hunter01` 投 `401` | 禁止当事人投票 |
| 重复投票 | `jury01` 再投 `401` | 提示已投过票 |

**内容安全调试**

建议测试这些关键词：

| 输入内容 | 预期 |
|---|---|
| “代写论文”“整篇论文”“替考” | 高风险/拦截 |
| “私下转账”“加微信付款” | 进入复核/风险提示 |
| “辱骂”“威胁”“人身攻击” | 拦截 |
| 普通跑腿/辅导/设计描述 | 通过或低风险 |

测试位置：
- 发布任务页：AI 风险评估、发布后的后台安全状态。
- 后台 `/admin/tasks`：安全状态、原因 tooltip、通过/驳回。
- 契约详情：提交证据说明。
- 小法庭详情：陈述、证据说明。
- 评价弹窗：安全提示展示。

**文件上传调试**

重点测三类：
- 正常图片：`png/jpg/webp`，应上传成功并可预览。
- 超过大小：默认单个不超过 `8MB`，应前端直接拦截。
- 类型不匹配：例如只允许图片时传 PDF，应提示类型不支持。

后台 `/admin/ops-config` 可调整：
- 单文件大小上限。
- 允许 MIME 类型。
- AI 内容安全开关。
- 陪审员最低信誉分。

**后台治理调试**

用 `admin` 逐页测：

1. `/admin/dashboard`：指标是否加载。
2. `/admin/users`：封禁、解封用户。
3. `/admin/certifications`：先用 `newbie` 提交认证，再用 admin 通过/驳回。
4. `/admin/tasks`：审核 `103`、`109`，检查安全标签。
5. `/admin/court-cases`：裁决 `401`。
6. `/admin/ops-config`：修改关键词、上传限制、投票门槛，保存后刷新检查是否保持在当前 mock 会话。
7. `/admin/audit-logs`：检查审核、封禁、裁决、配置更新是否留下记录。

**每天最小回归清单**

最少测这些就能判断前端有没有坏：

1. `npm run build` 通过。
2. `client01` 能登录、发布页能进。
3. `newbie` 进发布页会被拦到认证。
4. `hunter01` 能申请已发布任务。
5. `client01` 能选中申请并生成契约。
6. `hunter01` 能提交履约证据。
7. `client01` 能确认完成。
8. 双方能评价。
9. `jury01` 打开小法庭能看到投票区或重复投票提示。
10. `admin` 能打开 `/admin/tasks`、`/admin/ops-config`、`/admin/audit-logs`。

**常见坑**

- 一定确认浏览器打开的是 mock 服务地址，比如 `http://127.0.0.1:5174/`，不是旧的 `5173`。
- 切换账号前最好点退出登录；如果状态怪，清一下浏览器 localStorage 再登录。
- mock 数据不是数据库，刷新/重启后新增数据丢失是正常的。
- 如果你在 IDE 里看到 `.js` sidecar 残留，那不是前端源码入口；现在 `tsconfig` 已加 `noEmit: true`，正常不应再生成。
